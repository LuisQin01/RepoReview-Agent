# RepoReview Agent：框架、官方文档与源码清单

> 目标：学会判断框架什么时候有价值，而不是为了“项目用了框架”而重构。当前生产主线仍是 M8；本清单只用于学习与独立小实验。

## 一、先做决定

- [ ] 当前**不使用 LangChain/LangGraph 重构项目**。
- [ ] 当前**不需要阅读 LangChain 全部源码**。
- [ ] 当前优先读本项目自己的 controller、tool、state、provider、validation 和 Eval；这是面试最有价值的源码。
- [ ] 完成 M8 后，如出现“持久化恢复、人工审批、长运行任务、复杂状态图”中的真实需求，再在独立实验中评估 LangGraph。

## 二、必须看的官方文档

### A. OpenAI API：与本项目 function calling 最直接相关

| 顺序 | 文档主题 | 需要理解 | 动手验证（只用 mock） |
| --- | --- | --- | --- |
| 1 | Function Calling | 模型提出调用、应用本地校验与执行、再回传工具结果的完整循环；模型不能获得权限。 | 构造合法参数、坏 JSON、未知工具、超时、空响应。 |
| 2 | Structured Outputs | JSON schema/typed output 降低格式不确定性，但不保证事实正确。 | 为 hypothesis 和 verdict 写 schema；测试部分非法项。 |
| 3 | Evals | 固定输入、版本、配置和逐样例结果；质量与成本/延迟共同报告。 | 比较 single/verify，输出 TP/FP/FN/FPR 与 token。 |
| 4 | Rate limits / errors | 可重试错误、不可重试错误、超时、退避与预算。 | 使 provider mock 返回 timeout/错误码，验证不会伪装成功。 |

入口：访问 [OpenAI API 文档](https://developers.openai.com/api/docs)，在 Guides 中按上述标题阅读；以当前页面为准，不背会变化的模型名、价格和限额。

### B. LangChain：只学“工具、结构化输出、执行约束”

| 优先级 | 官方页 | 必须读的部分 | 读完要回答 |
| --- | --- | --- | --- |
| P0 | [Tools](https://docs.langchain.com/oss/python/langchain/tools) | tool 定义、schema、结构化返回、runtime state | 为什么“模型会调用”不代表工具应有权限？本项目 dispatcher 已替你做了哪些事？ |
| P0 | [Structured output](https://docs.langchain.com/oss/python/langchain/structured-output) | response format、provider/tool strategy、schema error handling | 为什么 M8 仍要本地 parser/validator，不能信任 provider schema？ |
| P1 | [Middleware overview](https://docs.langchain.com/oss/python/langchain/middleware/overview) | logging、retry/fallback、early termination、guardrail | 哪些是通用 middleware，哪些必须留在项目的确定性安全边界？ |
| P2 | Agent quickstart / `create_agent` | 只了解它封装了什么 | 它会隐藏哪些 controller 细节，为什么当前不适合直接替换？ |

**不必读**：集成目录、全部模型适配器、LangSmith 部署、Deep Agents、多 Agent 示例。它们不能帮助你完成 M8。

### C. LangGraph：只学“状态图与需要时的持久化”

| 优先级 | 官方页 | 必须读的部分 | 最小练习 |
| --- | --- | --- | --- |
| P0 | [Overview](https://docs.langchain.com/oss/python/langgraph/overview) | State、长运行工作流、durable execution 的问题定义 | 画出本项目 M8 的 state/node/edge，不写生产代码。 |
| P0 | [Graph API](https://docs.langchain.com/oss/python/langgraph/graph-api) | State、Node、Edge、conditional edge、state update | 用 TypedDict 建一个 `generate → collect → verify` 玩具图。 |
| P0 | [Workflows and agents](https://docs.langchain.com/oss/python/langgraph/workflows-agents) | workflow（路径固定）与 agent（工具选择动态）的区别 | 写一段话说明 M8 为什么首先是 workflow。 |
| P1 | [Persistence](https://docs.langchain.com/oss/python/langgraph/persistence) | checkpoint、thread、恢复、重放 | 内存 checkpointer 模拟一次暂停/恢复；对比本项目 trace。 |
| P1 | [Human-in-the-loop](https://docs.langchain.com/oss/python/langchain/human-in-the-loop) | interrupt、approve/edit/reject、checkpointer 前提 | 只为“发布 GitHub 评论”设计虚拟审批，不接生产。 |
| P2 | [Functional API](https://docs.langchain.com/oss/python/langgraph/functional-api) | 何时保留普通 Python 控制流 | 把玩具两轮流程改写一次，比较可读性。 |

LangGraph 的定位是状态化/长运行编排，不是让普通两轮流程自动更“智能”。只有当 checkpoint、恢复或人工审批能解决已验证痛点时，才值得局部引入。[官方概览](https://docs.langchain.com/oss/python/langgraph/overview)

### D. pytest：本项目真正每天使用的框架

- [ ] 阅读 pytest 官方文档的 `fixtures`、`parametrize`、`monkeypatch`、`raises`、`tmp_path`。
- [ ] 理解 mock provider 和 fake collector 的区别：前者模拟外部 LLM/provider，后者模拟可替换领域依赖。
- [ ] 能写一条“修复前失败、修复后通过”的回归测试；不要只测试函数没崩溃。

## 三、源码阅读的顺序与边界

### 第一优先级：你自己的项目

| 文件 | 只需要理解的关键点 | 读完证明 |
| --- | --- | --- |
| `src/review_service.py` | `ReviewRequest`、`ReviewService.review()`、fixed/react 分流、默认兼容性 | 画出从请求到 `ReviewResult` 的调用链。 |
| `src/agent_state.py` | `ReviewState` 的字段与写入者 | 解释 state 和 trace 的区别。 |
| `src/review_tools.py` | `ToolResult`、`ToolDispatcher`、三个工具、`finish_review` | 指出 schema、路径、敏感文件、截断检查的位置。 |
| `src/react_controller.py` | `ReActBudget`、loop、终止与 guardrail | 解释无限循环如何被阻止。 |
| `src/model_protocol.py` | `ToolCall`、`ModelResponse` 的内部协议 | 解释为何不传 SDK 对象。 |
| `src/llm_client.py` | mock、真实 provider 适配、错误映射 | 解释为何 SDK 只应出现在此边界。 |
| `src/validation.py` / `src/reporter.py` | finding gateway | 说明未来 M8 confirmed 如何复用它。 |
| `src/trace.py` / `src/eval_runner.py` | 脱敏审计、可复现指标 | 能说出一次实验最少记录哪些元数据。 |

### 第二优先级：只定点读框架源码

每项限时 60–90 分钟；达到“能画调用图 + 能提出一个对自己项目有意义的测试”即停止。

- [ ] LangGraph：`StateGraph` 的 build/compile、state update、conditional edge。
- [ ] LangGraph：内存 checkpointer 的 snapshot 与 resume 语义；不要先看数据库 saver。
- [ ] LangChain：tool 装饰器如何生成参数 schema，tool invocation 如何传播验证错误。
- [ ] LangChain：structured-output 的 provider strategy 与 tool strategy 如何处理 schema error。

**不要做**：通读 LangChain 仓库、背内部类名、复制框架源码进项目、因学习源码而替换 M8 的既定协议。

## 四、一个唯一的框架实验（M8 后再做）

- **题目**：用 mock provider 和 fake collector，在独立目录/分支复刻 M8 的 `generate_hypotheses → collect_evidence → verify_verdicts`。
- **对比项**：state 表达、空短路、预算、测试可读性、trace、暂停/恢复、依赖与迁移成本。
- **禁止项**：不替换生产 controller；不接真实模型、LangSmith、数据库、多 Agent。
- **产出**：一页 ADR，结论仅可为“当前不迁移”或“在 X 需求成立时局部采用”。
