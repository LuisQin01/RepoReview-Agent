下面沿用你当前 Codex 界面里的 **Sol / Terra / Luna** 命名，并使用"中 / 高"推理强度。OpenAI 官方目前将 Codex定位为实现、测试和代码审查工具；公开资料没有给出 Sol、Terra、Luna 的正式能力排名，因此这里是根据任务风险和 Token 性价比进行分配。

## 总体使用原则

- **Luna 中**：机械性、边界非常窄、已有实现模式可照搬。
- **Terra 中**：普通实现、Prompt 拼装、测试补充、配置接线。
- **Terra 高**：严格解析、注册分发、Provider 串联、状态与错误处理。
- **Sol 高**：协议设计、AST 语义、控制器、预算、降级、最终 Finding 链和里程碑验收。
- **独立 Review**：完成目标模式后，新开窗口使用 Review 模式，避免实现上下文带来的确认偏差。
- **常规 Review**：可以在原窗口切换 Review 模式，但必须只审查本任务 diff。
- **跳过 Review**：只是不启动 Review 模式，测试、diff 自检和交付记录仍然不能跳过。M8 的通用 DoD 本身要求正常、边界、失败路径测试以及完整测试结果。

### M8-15 跳过 Review 的条件

只有同时满足以下条件，M8-15-empty-hypothesis-short-circuit.md 才可以不启动 Review 模式：

1. 只修改控制器和对应测试，最多 2 个核心文件。
2. 不修改协议模型、parser、collector、validation 和公共 API。
3. 明确断言第二轮 provider 调用为 0、collector 调用为 0。
4. 合法空列表与坏 JSON 仍保持不同状态。
5. 聚焦测试和完整测试全部真实通过。
6. `git diff` 中没有顺手重构或其他任务内容。

任一条件不满足，就使用 **Terra 高 Review**。

### 你的清单中存在一个依赖矛盾

M8-19-multiround-trace.md 明确依赖 M8-18B-token-evidence-budget.md，但"最小可验收版本"却省略了 M8-18A-llm-call-budget.md、M8-18B-token-evidence-budget.md，直接列出 M8-19-multiround-trace.md。 

建议不要按省略后的顺序硬做，采用：

```text
M8-17-confirmed-finding-validation.md → M8-18A-llm-call-budget.md → M8-18B-token-evidence-budget.md → M8-19-multiround-trace.md → M8-20-single-verify-mode-switch.md
```

否则 M8-19-multiround-trace.md 中的预算统计和预算终止 trace 没有稳定来源。另一个可行方案是修改 M8-19-multiround-trace.md 的任务卡，把第一版范围明确缩减为"不包含预算统计"，但不建议在实现过程中临时绕过依赖。
