# 学习计划

你现在最该做的不是继续“学 Agent”，而是围绕一个能投简历的主项目，把 Hello-Agents 前九章的知识压成工程能力。

我建议从 **今天 2026-06-10 到 2026-07-10**，主线只做一件事：

> **RepoReview Agent：一个本地优先的 Code Review Agent**

它比单纯 LeetCode 复盘 Agent 更适合找实习；LeetCode 复盘可以作为它的一个小场景，而不是主项目。

我刚核对了当前几个方向：OpenAI Agents SDK 把 Agent 能力拆成 agent loop、tools、handoffs、guardrails、sessions、tracing 等工程原语；Anthropic 也强调从简单、可组合模式开始，不要一上来堆复杂框架；LangGraph 的重点是 durable execution、memory、human-in-the-loop 和 tracing；真实开源项目里，PR-Agent、OpenReview、Open SWE 都说明 **coding / review agent 是非常贴近真实岗位的方向**。参考：[OpenAI Agents SDK](https://openai.github.io/openai-agents-python/)、[Anthropic Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents)、[LangGraph Docs](https://docs.langchain.com/oss/python/langgraph/overview)、[MCP Docs](https://modelcontextprotocol.io/docs/getting-started/intro)、[PR-Agent](https://github.com/The-PR-Agent/pr-agent)、[OpenReview](https://github.com/vercel-labs/openreview)、[Open SWE](https://github.com/langchain-ai/open-swe)。

**最终目标**

到 2026-07-10，你要有这些东西：

1. 一个 GitHub 项目：`reporeview-agent`
2. 一个可运行 CLI：输入 `git diff`，输出结构化 code review
3. 支持仓库上下文读取，而不是只看 diff
4. 有 tool calling / 文件工具 / 结构化输出 / trace 日志
5. 有 15-20 条 eval case
6. README 里有架构图、demo、失败案例、改进方向
7. 简历上能写出一段具体项目经历
8. 面试时能讲清楚 Agent loop、RAG/context、tools、eval、guardrails

**每天固定节奏**

你最低每天 2 小时，我建议这样切：

```text
10 分钟：写下今天唯一目标
25 分钟：看文档/教程/源码，只看今天要用的部分
60 分钟：自己实现或改造一个小功能
15 分钟：调试、跑 demo
10 分钟：写复盘：今天做了什么，哪里坏了，怎么修的
```

如果当天能学 4-5 小时，加成这样：

```text
2 小时：Agent 主项目
1 小时：Hello-Agents / 官方文档对照学习
1 小时：LeetCode / 算法恢复
30 分钟：README、复盘、面试表达
```

重点：**每天必须留下代码或文档，不允许只看教程。**

**第 1 周：从跟敲切到独立实现**

时间：2026-06-10 到 2026-06-14  
因为今天是周三，我建议第一周先连续学 5 天，把节奏启动起来。

目标：做出最小版 `RepoReview Agent`。

**Day 1：确定项目骨架**

任务：

1. 建 GitHub 仓库或本地目录。
2. 写 `README.md` 第一版。
3. 明确项目一句话：

```text
RepoReview Agent 是一个本地代码审查 Agent，读取 git diff 和相关文件上下文，输出结构化 review 建议。
```

目录建议：

```text
reporeview-agent/
  src/
    cli.py
    diff_parser.py
    file_context.py
    reviewers.py
    schemas.py
    trace.py
  examples/
  evals/
  docs/
  README.md
```

验收标准：

```text
能用一句话讲清楚项目解决什么问题。
README 里有项目目标、输入输出示例、未来计划。
```

**Day 2：实现 diff 输入**

任务：

1. 支持读取 `git diff` 输出。
2. 解析出变更文件名、增删行、patch 内容。
3. 暂时不用 LLM，先打印结构化结果。

输出格式示例：

```json
{
  "files": [
    {
      "path": "src/auth.py",
      "added_lines": 12,
      "deleted_lines": 3,
      "patch": "..."
    }
  ]
}
```

验收标准：

```text
给一段 diff，程序能识别改了哪些文件。
```

**Day 3：实现文件上下文读取**

任务：

1. 根据 diff 文件路径读取原文件。
2. 支持限制最大读取字符数。
3. 如果文件不存在，要有友好错误。
4. 记录读取了哪些文件。

你今天学到的 Agent 点：

```text
工具不是随便调用的，要有权限、范围、失败处理和日志。
```

验收标准：

```text
输入 diff 后，程序能读取对应文件上下文。
```

**Day 4：做 rule-based reviewer**

先不急着接 LLM，先写规则检查器。

检查项：

1. 是否改了测试但没改业务代码。
2. 是否改了业务代码但没有测试。
3. 是否出现 `TODO`、`print`、`debugger`。
4. 是否可能缺少异常处理。
5. 是否有明显硬编码 key / token 字样。

验收标准：

```text
不用 LLM，也能输出 3-5 类基础 review。
```

**Day 5：接入 LLM reviewer**

任务：

1. 构造 prompt。
2. 输入 diff + 文件上下文 + rule findings。
3. 要求模型输出 JSON。
4. JSON 字段固定：

```json
{
  "findings": [
    {
      "severity": "high",
      "file": "src/auth.py",
      "line": 42,
      "issue": "...",
      "reason": "...",
      "suggested_fix": "...",
      "confidence": 0.78
    }
  ]
}
```

验收标准：

```text
能跑通一次完整 review：diff -> context -> reviewer -> JSON -> markdown。
```

**第 2 周：把 demo 做成 Agent 工程**

时间：2026-06-17 到 2026-06-21  
每周学 5 天，中间两天用于休息、补漏、复盘。

目标：加入 tool calling、trace、结构化校验。

**Day 6：设计 Agent loop**

你要写清楚这几个状态：

```text
1. receive_task
2. parse_diff
3. collect_context
4. run_static_checks
5. run_llm_review
6. validate_output
7. render_report
8. save_trace
```

不要直接上复杂框架，先用 Python 函数组装。

验收标准：

```text
README 有一张流程图或伪代码。
```

**Day 6 之后：求职优先版执行路线**

从这里开始，后续任务按“找 Agent / LLM 应用开发日常实习”的优先级重新排序。原来的 Day 7 之后计划可以作为参考，但实际执行优先看本节。

核心目标不是继续堆功能，而是把项目打磨成一个面试官能看懂、能运行、能评估、能追踪的 Agent 工程项目。

优先级：

```text
1. 显式 Agent State：说明 Agent 当前知道什么、下一步基于什么执行。
2. 结构化输出校验：LLM 输出不可信，必须校验。
3. 真实 LLM Client + Mock 模式：项目既能真实调用模型，也能离线测试。
4. Trace 落盘：每次运行都能复盘 Agent 做了什么。
5. Eval Case + Ground Truth：证明项目不是只会跑 demo。
6. 自动化测试：证明核心模块稳定。
7. README / Demo / Failure Analysis：让项目能用于求职展示。
```

**Day 7：显式设计 Agent State**

目标：

```text
把现在分散在 run_review_agent 里的变量，收拢成一个 ReviewState。
这一步能让项目更像 Agent 工程，而不是普通脚本。
```

任务：

1. 新建 `src/agent_state.py`，或者在 `src/schemas.py` 中增加 `ReviewState`。
2. `ReviewState` 至少包含：

```text
diff_path
repo_root
output_format
use_llm
diff_text
changed_files
contexts
rule_issues
llm_issues
issues
output
trace_steps
errors
```

3. 修改 `run_review_agent(args)`：先根据 `args` 创建 `state`。
4. 后续每一步都更新 `state`，例如：

```text
receive_task -> state.diff_path / state.repo_root
parse_diff -> state.diff_text / state.changed_files
collect_context -> state.contexts
run_static_checks -> state.rule_issues / state.issues
run_llm_review -> state.llm_issues / state.issues
render_report -> state.output
```

5. 暂时不用把每一步都拆成单独函数，但代码里要能看出 state 是主线。

验收标准：

```bash
python -m py_compile src\cli.py src\agent_state.py
python -m src.cli --diff examples\simple.diff --repo . --format markdown
python -m src.cli --diff examples\simple.diff --repo . --format markdown --llm
```

面试要能回答：

```text
我的 Agent State 是 ReviewState。
它记录任务输入、解析后的 diff、上下文、规则发现、LLM 发现、最终输出、trace 和 errors。
每个 step 都是在读取并更新这个 state。
```

**Day 8：结构化输出校验**

目标：

```text
LLM 输出坏 JSON、缺字段、字段类型错误时，程序不能直接崩溃。
```

任务：

1. 新建 `src/validation.py`。
2. 定义 `validate_llm_response(response_text)`。
3. 校验 JSON 是否能解析。
4. 校验顶层是否有 `findings`。
5. 校验每个 finding 是否包含：

```text
severity
file
line
issue
reason
suggested_fix
confidence
```

6. 允许轻度修复：

```text
line 缺失 -> 0
confidence 缺失 -> 0.5
severity 不合法 -> medium
file 缺失 -> "(unknown)"
```

7. 严重错误写入 `state.errors`，不要让程序崩溃。
8. 在 `llm_reviewer.py` 里调用 validation，而不是直接 `json.loads` 后就相信结果。

验收标准：

```text
正常 JSON：能输出 llm finding。
坏 JSON：程序不崩，trace/errors 中记录 llm_json_parse_error。
缺字段 JSON：程序不崩，能给默认值或记录 validation error。
```

建议准备 3 个手动测试样例：

```text
1. 正常 JSON
2. 少一个 suggested_fix 的 JSON
3. 完全不是 JSON 的字符串
```

面试要能回答：

```text
LLM 输出是不可信的，我把模型输出当成外部输入处理，进入系统前先做 schema validation。
```

**Day 9：真实 LLM Client + Mock 模式**

目标：

```text
项目既能真实调用模型，也能在没有 API Key 的情况下用 mock 模式跑通。
```

任务：

2. 实现两个函数：

```text

real_call_model(prompt)
```

3. 从环境变量读取 API Key。
4. CLI 增加参数：

```text

```

5. 默认使用 `mock`，避免别人 clone 项目后因为没有 key 直接失败。
6. 真实调用失败时，记录错误并降级，不让整个 review 崩溃。
7. 增加 `.env.example`，说明需要哪些环境变量。

验收标准：

```bash
python -m src.cli --diff examples\simple.diff --repo . --format markdown --llm --llm-provider mock
```

如果没有 API Key，运行真实 provider 时应该看到友好错误，而不是 Python traceback。

面试要能回答：

```text
我把 LLM 调用抽象成 llm_client，支持真实模型和 mock 模式。
这样方便测试，也方便未来替换模型供应商。
```

**Day 10：Trace 日志落盘**

目标：

```text
每次运行后，都能看到 Agent 执行了哪些 step、是否调用 LLM、发现了多少问题、有没有错误。
```

任务：

1. 完善 `src/trace.py`。
2. 实现 `save_trace(state, trace_dir="traces")`。
3. 每次运行生成：

```text
traces/{timestamp}_{task_id}.json
```

4. trace 至少包含：

```json
{
  "task_id": "...",
  "steps": [],
  "input_files": [],
  "context_files": [],
  "llm_called": true,
  "findings_count": 3,
  "duration_ms": 1234,
  "errors": []
}
```

5. CLI 增加：

```text
--trace
--trace-dir traces
```

6. trace 中不要保存完整敏感代码内容，先保存路径、数量、长度、错误摘要。

验收标准：

```bash
python -m src.cli --diff examples\simple.diff --repo . --format markdown --llm --trace
```

运行后 `traces/` 目录下出现 JSON 文件。

面试要能回答：

```text
trace 的作用是调试、解释和评估 Agent 行为。
如果输出质量不好，我能知道问题发生在 diff parsing、context retrieval、LLM review 还是 validation。
```

**Day 21：上下文预算和函数级上下文**

目标：

```text
让 Agent 不只是读取整个文件前 N 个字符，而是尽量读取和变更行最相关的函数、类、导入和调用关系。
```

为什么要做：

```text
PR-Agent 的一个核心能力是动态上下文：上下文太少会漏报，上下文太多会浪费 token 并干扰模型。
你的项目可以做一个轻量版本：先做到 Python 函数/类边界，再做到 import 和调用关系补充。
```

任务：

1. 新增 `src/context_budget.py` 或在 `file_context.py` 中拆出预算逻辑。
2. 定义上下文预算对象：

```text
max_total_chars
max_file_chars
max_extra_files
changed_file_priority
related_file_priority
```

3. 对 Python 文件使用 `ast` 找函数和类边界。
4. 根据 changed line 找到所在函数或类。
5. 上下文内容优先级：

```text
1. 被修改的 hunk
2. hunk 所在函数/类
3. 文件顶部 import
4. 同文件内被调用的函数
5. 通过 import 发现的本地相关文件
```

6. 给 `FileContext` 增加 metadata：

```text
context_type: changed | related | symbol
reason: direct_change | import_dependency | call_definition | same_symbol
truncated
chars_read
```

7. 增加上下文截断说明，不要静默丢弃内容。
8. 对非 Python 文件保留当前简单读取策略。
9. 增加测试：

```text
修改函数中间一行时，context 包含完整函数。
修改 class 方法时，context 包含 class 名称和方法体。
超过 max_total_chars 时，优先保留 changed file。
越权路径仍然被拒绝。
```

10. README 增加一段 `Context Strategy`。

验收标准：

```text
给定一个 300 行 Python 文件，修改某个函数内部一行时，报告上下文能定位到该函数。
context metadata 能解释每个文件为什么被读取。
总上下文不会超过预算。
相关文件读取失败时不会中断整个 review。
```

面试要能回答：

```text

为什么要有 context budget？
你如何决定哪些文件和哪些代码片段优先进入 prompt？
```
