# 新增里程碑实施清单：M7 与 M8

> 本文是后续代码实现的工作清单，而不是功能愿望列表。一次开发只选择其中一个编号任务；完成该任务的代码、测试和证据后，才进入下一个任务。

## 0. 使用规则与当前锚点

### 0.1 两个里程碑的目标与边界

| 里程碑 | 要解决的问题 | 最终产物 | 主要指标 | 不应混入的目标 |
| --- | --- | --- | --- | --- |
| M7：受控 ReAct 工具循环 | 固定上下文不足时的漏报，尤其是跨文件问题 | 在预算内由模型选择受限只读工具的审查模式 | Recall、跨文件 case 命中率、调用数、token、时延 | 多 Agent、任意代码执行、无限探索 |
| M8：假设—证据—验证审查 | 单轮模型把“猜测”直接报为 finding 的误报 | 固定两阶段的、可审计的验证模式 | Precision、FPR、Recall 变化、inconclusive 比例 | 自由 ReAct 循环、完整静态调用图 |

M7 和 M8 都依赖既有的上下文、Finding 校验、预算与离线 Eval 能力。它们可以复用底层安全与数据结构，但 **M8 不能直接复用 M7 的自由循环控制器**：M7 的路径由模型选择，M8 的路径必须固定为“两轮模型调用 + 确定性证据收集”。

### 0.2 实施前必须核验的现有锚点

当前仓库已经能看到下列相关模块；实现每一轮前仍须以代码和测试为准，不假设某个接口已完全满足需求。

| 能力 | 当前锚点 | 后续复用原则 |
| --- | --- | --- |
| 运行状态 | `src/agent_state.py` 的 `ReviewState` | 为新模式增加最小字段，不复制一份平行状态 |
| 领域模型与预算 | `src/schemas.py` 的 `ReviewIssue`、`FileContext`、`PythonSymbol`、`ContextBudget` | 新协议使用独立数据类；最终 finding 仍回到 `ReviewIssue` |
| Diff、上下文与符号 | `src/diff_parser.py`、`src/file_context.py` | 优先包一层适配器，禁止为工具层重写已有解析逻辑 |
| LLM | `src/llm_client.py`、`src/llm_reviewer.py` | provider 原始响应必须转换成内部结构，控制器不依赖 SDK 对象 |
| Finding 校验与输出 | `src/validation.py`、`src/reporter.py` | 所有最终 finding 统一走同一条校验、去重和渲染链 |
| Trace 与 Eval | `src/trace.py`、`src/eval_runner.py`、`evals/cases/` | 指标和 trace 仅记录可审计摘要，绝不记录敏感源码或完整内部推理 |

### 0.3 每个任务的通用 Definition of Done

除非任务卡另有更严格要求，完成一个任务必须同时满足：

1. 只实现任务卡所述范围；新增 API 有类型、docstring 或等价的可读契约。
2. 新增行为有聚焦的离线测试，至少覆盖正常路径、边界路径和指定的失败/降级路径。
3. 不调用真实 OpenAI 或 GitHub；涉及 provider 的测试必须 mock SDK 或使用脚本化 mock provider。
4. 不降低已有断言、不用 case 名称或固定输出“骗过” Eval、不把异常吞掉后伪装成成功。
5. 运行 `py -m pytest tests/ -v`；涉及评测时再运行 `py -m src.eval_runner --cases evals/cases --repo .`，记录真实输出。
6. 记录：变更文件、测试命令与结果、未验证项、失败/降级语义、下一个可独立完成的任务。

### 0.4 任务卡阅读方式

- **目标**：这一轮结束后可观察到的行为，而不是实现手段。
- **任务**：本轮唯一允许的代码与测试工作。
- **非目标**：明确禁止的范围扩张。
- **验收与证据**：必须能够通过测试、命令输出或机器可读产物证明的事实。
- **依赖**：未满足时应先做前置任务，不能靠猜测继续实现。

### 0.5 跨任务的实现契约

以下约定适用于 M7、M8 的所有新增代码。任务卡未重复说明时，仍必须遵守；如果现有代码的语义与此冲突，先补充或修正前置任务，不在控制器中临时绕过。

| 主题 | 必须遵守的契约 |
| --- | --- |
| 路径与范围 | 所有面向模型的路径均为相对 `repo_root` 的规范化路径；任何绝对路径、`..` 穿越、符号链接越权、敏感路径和 review scope 外路径都必须在工具/collector 边界被拒绝。 |
| 对外数据 | controller、工具和 collector 之间只传递内部数据类或可 JSON 序列化的字典；不得传递 OpenAI SDK 对象、文件句柄、异常对象或可执行回调。 |
| 错误语义 | 预期失败必须返回稳定状态码，而不是依赖异常消息字符串。建议统一使用：`invalid_arguments`、`forbidden`、`not_found`、`unavailable`、`truncated`、`unsupported`、`internal_error`；`internal_error` 的诊断详情只可进入受脱敏保护的日志。 |
| 结果与截断 | 每个可变长度结果必须同时携带实际/上限大小和 `truncated` 标记。截断表示“信息不完整”，绝不等价于“未找到”或“证据支持”。 |
| 身份与关联 | 工具调用、假设、证据请求和 verdict 都要有稳定、可追踪的 ID；关联只能按 ID 建立，禁止依赖列表位置、自然语言文本或模型返回顺序。 |
| 安全与可复现 | 单元测试及 Eval 默认离线、确定性执行；网络、真实 provider、任意 shell/代码执行均不属于 M7/M8 的允许能力。时间、随机性和预算应可由测试注入或固定。 |
| Finding 唯一入口 | 只有通过既有 `validation`、去重、排序与 `reporter` 链的 `ReviewIssue` 才能出现在最终报告。工具文本、假设、证据和 `rejected`/`inconclusive` verdict 绝不能旁路成为 finding。 |

### 0.6 任务开始、完成与移交

每张任务卡在编码前应满足 **Definition of Ready**：依赖任务已通过；涉及的现有 API 已由源码和测试确认；输入、输出、失败语义及预算归属均已写入测试名或任务记录。若其中一项不清楚，任务状态应为“待澄清”，而不是以实现猜测替代设计。

任务完成时，除 0.3 的通用 DoD 外，必须留下一个与任务 ID 对应的简短移交记录。记录可以放在 PR 描述、提交说明或专用的机器可读结果文件中，但至少包含下表字段：

| 字段 | 内容 |
| --- | --- |
| `task_id` / `commit` | 例如 `M7-3` 与实现所对应的提交；未提交时明确标注工作区状态。 |
| `changed_surfaces` | 修改的生产代码、测试、Eval case、配置和文档路径。 |
| `contract` | 新增或改变的输入、输出、错误码、默认值和兼容性影响。 |
| `verification` | 实际执行的命令、通过/失败结果及关键指标；不可把“预期通过”当结果。 |
| `failure_semantics` | 预算耗尽、超时、解析失败、无结果等情况对最终 findings 与 trace 的影响。 |
| `follow_up` | 尚未验证的事项，以及下一个依赖已满足、可独立执行的任务。 |

### 0.8 里程碑级评测与准入规则

任务卡的单元测试通过，不等于里程碑可推广。M7-0/M8-0 的基线和 M7-18/M8-22 的对照必须使用**同一 commit、case 集、mock 脚本或模型版本、随机种子、预算配置和运行环境**。报告至少包含运行清单、逐 case 结果、聚合指标、失败原因和不可比较项。

| 决策 | 硬性准入条件 | 允许保留为“待优化”的内容 |
| --- | --- | --- |
| M7 可合入 | fixed 默认路径全量回归；所有路径/敏感信息/预算拒绝测试通过；至少一个跨文件 case 证明工具结果确实进入下一轮且 final finding 仍经既有校验。 | Recall、时延与成本尚未优于基线，但必须如实报告且 `react` 继续保持显式 opt-in。 |
| M7 可讨论默认启用 | 在代表性 case 集上不存在未解释的 Precision 或 Recall 回退；收益、p95 时延、调用数和 token/成本均有可复现对照。 | 具体阈值可在收集稳定基线后按产品目标设定，不能事后为单次结果调整。 |
| M8 可合入 | `rejected` 与 `inconclusive` 绝不产生最终 finding；空假设、证据不可用、预算耗尽和部分 collector 失败均有确定性测试；single 默认路径回归通过。 | 误报改善尚未达到业务目标，但 `verify` 必须保持显式 opt-in 并输出完整 verdict/失败摘要。 |
| M8 可讨论默认启用 | 负样例上的误报下降有证据，且 Recall 下降、inconclusive 上升、时延和成本均已量化并获接受。 | 需要更多 case 或人工标注时，结论为“证据不足”，不得宣称 Precision 已提升。 |

# 一、M7：受控 ReAct 工具调用循环

## M7 里程碑验收标准

M7 通过的条件是：在 `fixed` 模式保持默认行为不变的前提下，`react` 模式能够在明确的步骤、调用、token 与工具结果预算内，使用只读且受校验的工具完成审查；最终 finding 仍走既有校验、排序与报告链。离线对照 Eval 必须报告 Recall、Precision、F1、跨文件 case、平均/p95 时延、模型调用数、token/估算成本及失败原因。只有有证据证明收益与成本可接受，才讨论是否将其作为默认路径。

## M7-0：固定流水线基线

- **目标**：为后续所有 ReAct 结论建立可复现、可比较的 `fixed` 基线。
- **依赖**：现有 `pytest` 与 `eval_runner` 可运行。
- **任务**：核对固定流水线的入口、状态流与输出链；运行完整测试及离线 Eval；将 commit/配置、指标、调用次数和时延保存为机器可读基线结果；列出 M2/M3/M4 前置能力的“已确认/未确认”状态。
- **非目标**：不增加工具、模式开关、prompt 或模型调用。
- **验收与证据**：基线包含 Precision、Recall、F1、FPR、平均 finding 数、平均时延；记录可复制命令与输入配置；结果文件可由测试或脚本读取；未满足的前置条件有对应代码或测试证据。

## M7-1：工具协议与结果结构

- **目标**：定义 provider 无关、可序列化的只读工具契约。
- **依赖**：M7-0。
- **任务**：新增 `ReviewTool` 协议和 `ToolResult` 数据结构。`ToolResult` 至少包含成功状态、给模型的安全摘要、稳定错误码、是否截断、结果大小/usage；为工具声明 `name`、`description` 与 JSON Schema。
- **非目标**：不注册工具、不执行工具、不接入 LLM/function calling。
- **验收与证据**：fake tool 可满足协议；成功、预期失败和内部失败结果可区分；schema 与结果可 JSON 序列化；测试证明内部错误详情不会自动暴露给模型摘要。

## M7-2：工具注册表与 Dispatcher

- **目标**：让控制器能按工具名安全、确定地调用已注册工具。
- **依赖**：M7-1。
- **任务**：实现注册、查找、分发和工具异常转换；重复名字在注册时失败；未知名字返回稳定的结构化 `ToolResult` 错误。
- **非目标**：不接入模型、不做参数 schema 校验、不新增真实仓库读取工具。
- **验收与证据**：fake tool 的调用参数与返回值正确传递；重复注册被拒绝；未知工具和 tool 抛异常不会让 dispatcher 抛出未处理异常；测试不依赖文件系统或网络。

## M7-3：工具参数 Schema 校验

- **目标**：任何模型给出的参数在执行前都经过本地白名单校验。
- **依赖**：M7-2。
- **任务**：在 dispatcher 的执行边界使用每个工具的 schema 验证对象类型、必填字段、字段类型和禁止的额外字段；验证错误转为稳定错误码与安全摘要。
- **非目标**：不让 provider 修复参数，不新增工具或循环。
- **验收与证据**：缺少必填参数、类型错误和额外字段均失败；合法参数正常调用；通过 spy/fake 明确证明非法参数时 `run()` 为零次；错误中不回显敏感参数值。

## M7-4：`get_changed_hunks` 工具适配器

- **目标**：模型只能读取当前 review scope 内、保留新文件行号范围的变更块。
- **依赖**：M7-2、M7-3，以及现有 diff parser 的行为已由测试覆盖。
- **任务**：将 `ChangedFile`/hunk 解析结果适配为只读工具；指定文件返回受大小限制的 hunk 摘要与 `new_start/new_end`；超出 scope、缺失文件和无 hunk 使用结构化结果表示。
- **非目标**：不修改 diff parser，不读取完整仓库文件，不实现 ReAct controller。
- **验收与证据**：合法文件返回正确 hunk 与新行范围；多 hunk、重命名及带空格路径不丢失定位；超出 review scope 被拒绝；截断状态可见且不伪称内容完整。

## M7-5：`read_file_context` 工具适配器

- **目标**：在不突破现有安全边界和预算的前提下读取受限文件上下文。
- **依赖**：M7-3，以及 `file_context` 的安全、预算与敏感信息行为已核验。
- **任务**：把既有上下文读取能力包装为工具；统一处理 repo root、路径规范化、敏感文件、文件不存在和截断；返回面向模型的安全摘要及必要元数据。
- **非目标**：不自行实现第二套文件读取/脱敏规则，不搜索 symbol，不加入循环。
- **验收与证据**：只能读 repo root 内的允许文件；`..` 路径穿越与符号链接越权被拒绝；敏感文件不进入结果；大文件标记截断；不存在文件是可恢复结构化失败而非崩溃。

## M7-6A：`search_python_symbol` 工具适配器

- **目标**：把已有 Python AST 符号定位能力以受限工具形式提供给模型。
- **依赖**：M7-3、M7-5，以及现有 `PythonSymbol` 定位能力。
- **任务**：定义最小参数（受控文件/符号或变更行）和稳定结果；返回函数、方法、类的限定名与行范围，并受结果大小限制。
- **非目标**：不实现跨文件调用图，不支持任意语言的 symbol search。
- **验收与证据**：覆盖函数、类、方法、不存在符号、Python 语法错误、非 Python 文件和越权路径；解析失败不会转为“找不到”；截断与失败状态可区分。

## M7-6B：`list_imports` 工具适配器

- **目标**：在仓库范围内提供有限、可解释的 import 信息。
- **依赖**：M7-3、M7-5。
- **任务**：解析普通 import、`from ... import ...` 和相对 import；输出语句、候选仓库路径或“外部/不确定”状态。
- **非目标**：不构建完整 import 图、不无限递归解析、不把第三方包当作可读仓库文件。
- **验收与证据**：上述三种 import 被正确区分；语法错误、外部依赖和无候选目标有明确状态；返回路径都在 repo root 内；多候选不假装唯一。

## M7-7：`finish_review` 终止语义

- **目标**：定义唯一、可验证的正常终止入口，避免模型输出文本即被误当 finding。
- **依赖**：M7-1、既有 Finding validator。
- **任务**：定义终止调用的输入与状态变更：可否为空、重复调用语义、单条非法 finding 的过滤/拒绝策略；终止参数中的 finding 立即进入既有 schema/位置校验链。
- **非目标**：不实现整个 controller，不复制 Finding validation。
- **验收与证据**：合法结果终止；空结果可安全终止；非法 finding 不能进入最终输出；重复 finish 有明确且测试覆盖的策略；后续控制器可据此保证 finish 后不再执行工具。

## M7-8：脚本化 Mock Provider

- **目标**：离线、确定地模拟多轮工具调用和 provider 失败。
- **依赖**：M7-1、M7-7。
- **任务**：扩展 mock provider，使其按预设脚本逐次返回文本、工具调用、finish 或异常；脚本耗尽采用明确错误或安全结束语义；保留每次收到的请求以供断言。
- **非目标**：不改真实 provider，不实现控制循环。
- **验收与证据**：测试能模拟合法调用、非法工具名、非法参数、坏 JSON 和 provider 异常；每次调用只消耗一个脚本项；没有网络访问；脚本耗尽不产生伪造 finding。

## M7-9：统一 `ModelResponse` 与 `ToolCall`

- **目标**：控制器只依赖内部模型响应，而不依赖 OpenAI SDK 的原始对象。
- **依赖**：M7-8。
- **任务**：定义带稳定 ID、名称、解析后参数的 `ToolCall`，以及能表达文本、多个调用、finish/停止原因和 usage 的 `ModelResponse`；明确文本与调用同时出现时的处理规则。
- **非目标**：不执行工具、不接 SDK、不实现 controller。
- **验收与证据**：纯文本、单/多调用、finish、坏 arguments 与混合响应都有构造/解析测试；调用 ID 保留；非法 arguments 不会被静默替换为空字典。

## M7-10：真实 Provider 的 function calling 适配

- **目标**：真实 provider 可以接收内部工具 schema，并将响应转换为 `ModelResponse`。
- **依赖**：M7-9，现有 provider 的超时/重试语义已测试。
- **任务**：只在 provider 边界完成 schema 转换、SDK 响应解析、usage 映射和异常映射；保持固定流水线默认调用路径不变。
- **非目标**：provider 不执行工具、不验证/修复参数、不实现循环或 fallback provider。
- **验收与证据**：mock SDK 覆盖无调用、单调用、多调用、坏 JSON、timeout 与 provider 错误；转换结果符合 M7-9；固定模式的既有测试不变；测试绝不调用真实 API。

## M7-11：最小 ReAct Happy Path

- **目标**：以 mock provider 证明“模型 → 工具 → 历史 → finish”的闭环真实存在。
- **依赖**：M7-2、M7-7、M7-8、M7-9。
- **任务**：实现最小控制器，仅支持顺序执行一个或多个合法工具调用，将安全 `ToolResult` 追加到下一轮历史，并在 `finish_review` 后返回校验后的 findings。
- **非目标**：不加最大步数、预算、重试、复杂失败恢复或真实 provider 集成。
- **验收与证据**：脚本化测试断言调用顺序；第二轮请求确实包含上一轮工具结果；最终 finding 来自 finish 参数而非硬编码；finish 后零次额外 provider/tool 调用。

## M7-12A：最大步骤上限

- **目标**：控制器在不正常终止时绝不会无限循环。
- **依赖**：M7-11。
- **任务**：增加 `max_steps` 配置、计数和明确终止原因；定义达到上限时如何保留已通过校验的结果并标记降级完成。
- **非目标**：不同时加入 token/调用预算。
- **验收与证据**：无限工具脚本在上限处停止；上限后 provider 调用次数不增加；结果/trace 记录 `max_steps_exhausted`；不会把预算终止伪报为正常 finish。

## M7-12B：调用、Token 与工具结果预算

- **目标**：既有预算在 ReAct 模式同样生效且可观察。
- **依赖**：M7-12A、`ContextBudget` 行为已核验。
- **任务**：引入或复用 `max_llm_calls`、累计 token 与单次/累计工具结果大小限制；预算状态写入 `ReviewState` 或等价 trace 状态。
- **非目标**：不改变 fixed 模式预算语义，不把全部预算失败变成无说明的空 findings。
- **验收与证据**：分别覆盖调用数耗尽、token 耗尽和超大工具结果截断；后续调用被阻止；截断/终止原因可被报告；正常任务不被误截断。

## M7-13：非法与失败工具调用策略

- **目标**：工具失败具有有限、统一且可观察的恢复策略。
- **依赖**：M7-11、M7-12A。
- **任务**：定义未知工具、参数非法、工具异常、超时、不可序列化结果以及同轮部分成功的处理规则；将有限错误摘要反馈给模型；连续失败达到阈值时终止。
- **非目标**：不无限重试，不用空列表伪装审查成功。
- **验收与证据**：每类失败有稳定错误码；单次失败不使整个 review 崩溃；连续失败止于阈值；成功的并列调用结果不被失败调用抹掉；主机绝对路径和敏感参数不会出现在摘要。

## M7-14：工具历史安全处理

- **目标**：进入模型历史与 trace 的工具结果均经过统一 guardrail。
- **依赖**：M7-5、M7-13，以及既有敏感信息测试。
- **任务**：在 history/trace 写入前统一执行敏感文件拒绝、秘密模式脱敏、长度截断与路径错误净化；复用 M3 guardrail，而非另建一套规则。
- **非目标**：不保存模型完整内部推理；不扩大可读取文件范围。
- **验收与证据**：`.env`、私钥、证书与 API key 模式不进入 history 或 trace；超长源码被截断并标记；越权错误不泄露宿主机路径；现有敏感信息回归测试保持通过。

## M7-15：ReAct State 与 Trace

- **目标**：为每一步提供足够审计信息，而不存储内部推理或敏感原文。
- **依赖**：M7-11、M7-14。
- **任务**：记录 step 序号、工具名、参数摘要、结果摘要、成功状态、时延、usage 与终止原因；使用 `decision_summary` 或 `model_rationale_summary` 等受控摘要字段。
- **非目标**：不记录 chain-of-thought，不改变控制器决策。
- **验收与证据**：正常 finish、预算终止和工具失败均有可解析 trace；摘要不包含完整源码/秘密；时延和调用计数可与 mock provider 断言对应；关闭 trace 时主流程行为不变。

## M7-16：`fixed` / `react` 模式开关

- **目标**：一个 review service 能显式选择模式，默认兼容现有固定流水线。
- **依赖**：M7-15。
- **任务**：在 CLI/config/state 增加受限的 `review_mode`（例如 `fixed`、`react`）；两种模式复用既有 validation、ranker、reporter 与安全边界；模式写入状态与报告元数据。
- **非目标**：不静默 fallback；不让 react 失败后被报告为 fixed 成功。
- **验收与证据**：无参数时仍为 fixed；非法模式值被拒绝；两种模式均可离线 smoke；react 失败有明确失败/降级状态；相同合法 finding 经两种模式的最终校验结果一致。

## M7-17：跨文件 Eval Case

- **目标**：新增一个只有补充仓库上下文才可可靠发现的真实问题样例。
- **依赖**：M7-0；不依赖 React 实现完成。
- **任务**：在 `evals/cases/` 创建输入 diff、必要仓库上下文与 ground truth；说明 fixed 为什么看不到关键证据，以及 ReAct 需要哪种工具，不允许为 case 写规则特判。
- **非目标**：不修改 controller/prompt 来让样例通过。
- **验收与证据**：ground truth 经人工核验；同一 case 可由 static、fixed LLM、react 使用；期望 finding 有文件/行/类别；case 名称改变不影响逻辑；测试能防止直接根据 case 名称输出答案。

## M7-18：Fixed 与 ReAct 对照 Eval

- **目标**：用固定、可复现评测而非主观印象判断 M7 是否有效。
- **依赖**：M7-16、M7-17。
- **任务**：在同一 commit、case 集、模型配置与预算下运行 fixed/react；按 case 输出差异并聚合指标；保留机器可读结果和配置。
- **非目标**：不在本轮调整实现、prompt 或 case 以追求分数。
- **验收与证据**：报告 Precision、Recall、F1、跨文件结果、平均/p95 时延、token、调用数、估算成本、未知工具/预算耗尽数；新增误报和修复漏报逐 case 可追溯；若无统计或证据支持，结论必须为“未验证”而非“提升”。

# 二、M8：假设—证据—验证审查

## M8 里程碑验收标准

M8 通过的条件是：`verify` 模式严格执行“生成结构化假设 → 确定性收集白名单证据 → 结构化 verdict → 既有 Finding 校验/排序”的固定协议。只有 `confirmed` 且通过既有校验的 finding 才能输出；`rejected` 与 `inconclusive` 绝不输出为 finding。离线对照 Eval 必须证明目标误报类别的 FPR/Precision 改善，并同时报告 Recall、token、调用次数、时延与证据失败率，防止以大量 `inconclusive` 换取表面低误报。

## M8-0：单轮 LLM 基线

- **目标**：固定当前 `single` 审查模式的误报和成本基线。
- **依赖**：现有 Eval 可运行；M7 不要求已完成。
- **任务**：用固定 case、配置和 commit 运行单轮模式，保存 Precision、Recall、F1、FPR、token、调用次数、时延与逐 case 结果。
- **非目标**：不实现假设、证据或第二轮调用。
- **验收与证据**：基线结果机器可读且含负样例统计；命令可复现；与 M7 基线分开标识模式和配置，不混合比较。

## M8-1：假设、证据与 Verdict 数据模型

- **目标**：为两轮协议建立强约束、低歧义的数据边界。
- **依赖**：M8-0、既有 `ReviewIssue` schema。
- **任务**：定义 `ReviewHypothesis`、`EvidenceRequest`、`EvidenceItem`、`VerificationVerdict`、`DialogueTraceEntry`；校验 hypothesis 的文件/行/描述/证据请求、合法 confidence，及 verdict 只能为 `confirmed`/`rejected`/`inconclusive`。
- **非目标**：不写 prompt、不调用模型、不用一个大量可空字段的 `DialogueTurn` 承担全部职责。
- **验收与证据**：confirmed 必须带合法候选 finding；rejected/inconclusive 不得带最终 finding；不支持 evidence type 与越界字段被拒绝；数据可安全序列化。

## M8-2：Evidence Type 白名单与参数 Schema

- **目标**：限定模型可请求的确定性证据，避免变成任意仓库探索。
- **依赖**：M8-1。
- **任务**：第一版仅登记 `symbol_definition`、`caller_exception_handling`、`import_target`，并为各自定义参数 schema、数量上限与去重键。
- **非目标**：不承诺完整 `call_chain`，不实现 collector。
- **验收与证据**：合法请求通过；未知 type、缺参、错误类型、超限和重复请求均按明确策略处理；去重键不依赖字典偶然顺序。

## M8-3：第一轮假设响应解析器

- **目标**：在调用 LLM 前后把不可信 JSON 严格转换为有限的有效假设集合。
- **依赖**：M8-1、M8-2。
- **任务**：解析顶层结构和 `hypotheses` 列表；验证字段、confidence、文件/行、证据类型及数量；定义部分无效时“保留合法项并记录错误”还是“全拒绝”的一致语义。
- **非目标**：不调用 provider，不收集证据。
- **验收与证据**：覆盖合法/空列表、非 dict 顶层、非 list、坏 JSON、缺字段、非法 confidence、未知 type、部分非法和超限；测试明确断言被拒绝项不会流入后续阶段。

## M8-4：假设生成 Prompt Builder

- **目标**：让第一轮只提出待验证假设，而不是直接声称最终 finding。
- **依赖**：M8-2、现有 diff/context 构建能力。
- **任务**：将 diff、预算内上下文与规则结果构造成 prompt；声明输入不可信、假设数量上限、evidence 白名单和禁止请求任意文件/执行代码；输出只允许 schema 所需字段。
- **非目标**：不执行第一轮调用，不写第二轮 prompt。
- **验收与证据**：测试检查关键约束、上下文预算、规则结果引用和用户输入隔离；不通过脆弱的整段 prompt 字符串比较；超长输入不会越过 prompt 预算。

## M8-5：第一轮假设生成调用

- **目标**：使用 mock provider 生成并返回已经解析的有限假设列表。
- **依赖**：M8-3、M8-4。
- **任务**：串联 builder、provider、parser 和状态/错误记录；保持 timeout/provider 异常的既有语义；将无效输出作为结构化失败而不是假设。
- **非目标**：不收集证据，不发第二轮调用。
- **验收与证据**：两条合法、空、坏 JSON、timeout、部分非法和超限场景全覆盖；返回的是 parser 产物；mock 可断言调用次数；空/失败不产生最终 finding。

## M8-6：Evidence Collector 协议与注册表

- **目标**：对每种证据类型建立可替换、可隔离失败的收集边界。
- **依赖**：M8-1、M8-2。
- **任务**：定义 `EvidenceCollector.collect(request, state) -> EvidenceItem` 和 registry/dispatcher；统一未知 type、collector 异常、超时、结果大小限制与去重策略。
- **非目标**：不实现具体 collector，不依赖 M7 controller。
- **验收与证据**：fake collector 覆盖注册、分发、未知 type、异常、超时、超大结果和重复请求；失败是结构化 `EvidenceItem`，不会让整次审查异常退出。

## M8-7：符号定义证据收集器

- **目标**：可靠提供给定 file/line/symbol 对应的 Python 定义证据。
- **依赖**：M8-6、既有 AST/上下文能力。
- **任务**：实现 `symbol_definition` collector，返回限定名、行范围、受预算限制的定义片段及来源；区分未找到、无法解析、非 Python 与越权。
- **非目标**：不搜索整个调用图，不根据模型推测补全定义。
- **验收与证据**：函数、类方法、未找到、语法错误、非 Python、越权、截断均有测试；`not_found` 不等于 `unavailable`；返回路径在 repo root 内。

## M8-8：调用方异常处理证据收集器

- **目标**：针对已知调用点，确定其是否位于相关的 `try/except` 中。
- **依赖**：M8-6。
- **任务**：第一版只接收已知 file 与 line，使用 AST 判断包围该调用的 `try`，记录捕获类型、bare except 和嵌套结构；证据只描述事实，不自行判断是否为 bug。
- **非目标**：不自动发现全仓库调用方，不构建完整调用图。
- **验收与证据**：覆盖匹配异常、无关异常、无处理、bare except、嵌套 try 与 AST 失败；不相关 except 不被误标为已处理；行号定位失败是 `unavailable`/`not_found` 而非 confirmed。

## M8-9：Import Target 证据收集器

- **目标**：将 import 映射为仓库内候选目标或明确的不确定状态。
- **依赖**：M8-6。
- **任务**：处理绝对/相对 import、模块不存在、外部依赖、循环 import 与多个候选；只返回 repo root 内候选路径。
- **非目标**：不递归无限解析 import 链，不读取外部包。
- **验收与证据**：各类型都有测试；多个候选返回 `inconclusive`；外部依赖不会误报“模块不存在”；越权路径被拒绝。

## M8-10：证据收集编排器

- **目标**：把多个假设的证据请求去重、执行并准确回填到对应假设。
- **依赖**：M8-5、M8-6，至少一个具体 collector 已完成。
- **任务**：构建去重请求计划，执行后按 hypothesis ID 分组回填；设置总请求数、总结果大小与上下文预算；单个 collector 失败只影响相关假设。
- **非目标**：不调用第二轮 LLM，不从第一轮假设直接生成 finding。
- **验收与证据**：同一请求只执行一次；一假设多证据、跨假设共享证据和部分失败均正确分组；预算超限有记录；其他假设仍可继续。

## M8-11：证据失败与 `inconclusive` 语义

- **目标**：明确“没有证据”与“系统无法获得证据”之间的关键差别。
- **依赖**：M8-10。
- **任务**：统一 `not_found`、`unavailable`、`conflicting`、`truncated`、`unsupported` 状态和各自的可验证含义；规定证据不足/失败绝不自动确认 finding。
- **非目标**：不新增模型调用，不把收集异常吞为肯定结论。
- **验收与证据**：每种状态有构造与编排测试；collector 异常不崩溃；无可用证据的 hypothesis 不能产生 confirmed finding；trace 可区分每种原因。

## M8-12：第二轮 Verdict 响应解析器

- **目标**：把验证轮的输出严格限制为逐 hypothesis 的三态 verdict。
- **依赖**：M8-1、M8-11、既有 Finding validator。
- **任务**：解析 verdict JSON；验证 hypothesis ID/index 覆盖与唯一性、状态合法性及 confirmed finding；在解析边界对 finding 使用既有 schema，而最终位置校验仍保持在统一校验链。
- **非目标**：不调用模型，不绕过/复制 finding validation。
- **验收与证据**：confirmed、rejected、inconclusive、越界 ID、重复 verdict、漏 verdict、非法 finding 与顶层类型错误都有测试；rejected/inconclusive 携带 finding 被拒绝。

## M8-13：验证 Prompt Builder

- **目标**：第二轮仅依据“原始假设 + 已收集证据”作出逐项 verdict。
- **依赖**：M8-10、M8-11、M8-12。
- **任务**：构建有稳定 hypothesis ID 的验证 prompt；标注证据状态和截断；明确不能引入证据外事实、每项必须返回 verdict、只有 confirmed 可给 finding。
- **非目标**：不执行控制器，不保存内部推理。
- **验收与证据**：测试验证 ID 与证据正确关联而非按列表位置猜测；缺失/截断证据显式出现；输出 schema 和数量约束存在；prompt 不超过配置预算。

## M8-14：两轮 Happy Path 控制器

- **目标**：离线证明两轮协议确实淘汰被拒绝假设，而非换个字段名直接输出第一轮结果。
- **依赖**：M8-5、M8-10、M8-12、M8-13。
- **任务**：使用脚本化 mock 串联“假设生成 → 证据收集 → 验证 → 最终校验入口”；只把第二轮 confirmed 的候选 finding 送入后续链。
- **非目标**：不加空短路、预算/失败降级或模式开关。
- **验收与证据**：两假设样例中一条 confirmed、一条 rejected，最终恰一条 finding；第二轮请求包含对应证据；调用顺序/次数可断言；第一轮文本自身不可能直接成为结果。

## M8-15：空假设短路

- **目标**：没有合法假设时零成本、正常结束。
- **依赖**：M8-14。
- **任务**：将第一轮空列表或全部无效的有效结果定义为短路完成；记录短路原因。
- **非目标**：不收集证据、不调用验证轮、不把空结果记为 provider 失败。
- **验收与证据**：第二次模型调用和 collector 调用均为零；返回合法空 findings；trace 显示 `no_valid_hypotheses`；坏 JSON 与合法空列表仍可区分。

## M8-16：证据失败降级

- **目标**：证据不可用时保守地输出 `inconclusive`，而非确认未经验证的假设。
- **依赖**：M8-11、M8-14。
- **任务**：规定：全部所需证据不可用时本地直接 inconclusive；部分可用时可以进入验证轮，但必须携带缺失状态；将该规则落实进控制器和 trace。
- **非目标**：不把“无法读文件”解释为问题存在，不重试到无限次。
- **验收与证据**：全不可用时无第二轮或按设计有明确零/一次调用；部分可用时验证 prompt 含缺失状态；最终未验证 finding 数为零；不同失败状态可追溯。

## M8-17：Confirmed Finding 接入既有校验链

- **目标**：多轮模式没有第二套、标准更低的 finding 输出路径。
- **依赖**：M8-14、既有 `validation` 与 ranker 行为。
- **任务**：将 confirmed candidates 送入同一 schema、changed-hunk/scope 校验、去重、排序与报告器；拒绝结果永不进入该入口。
- **非目标**：不在多轮 controller 复制 validation，不改变既有 rule/LLM finding 语义。
- **验收与证据**：行号不在 changed hunk、非法 severity、scope 外文件、重复 confirmed 均按既有语义处理；rejected 不会调用 validator；最终报告 provenance 可区分 verify 模式来源。

## M8-18A：两轮调用预算

- **目标**：在第一轮后额度不足时，安全地阻止第二轮。
- **依赖**：M8-14、现有调用预算或 M7-12B 的可复用预算机制。
- **任务**：在每次模型调用前检查剩余额度；定义无额度时所有未验证假设的终止/降级表示。
- **非目标**：不同时实现 token/evidence 预算，不绕过固定预算。
- **验收与证据**：第一轮消耗最后一次额度后，验证轮零调用；已有 confirmed finding 不被伪造；trace/报告记录 `llm_call_budget_exhausted`；fixed/single 模式行为不回归。

## M8-18B：Token 与 Evidence 预算

- **目标**：限制两轮审查的假设规模、证据规模和第二轮 prompt 大小。
- **依赖**：M8-18A、M8-10。
- **任务**：限制假设数、每假设证据数、总 evidence 字符数与第二轮 prompt；为截断证据和未处理假设保留显式状态。
- **非目标**：不把所有预算耗尽统一转为空 findings。
- **验收与证据**：分别覆盖正常完成、假设超限、证据总量超限、prompt 超限、部分验证与全部 inconclusive；最终统计与 trace 的计数一致。

## M8-19：多轮 Trace

- **目标**：从 trace 还原每个阶段的数量、状态、耗时与成本，而不泄露内部推理。
- **依赖**：M8-16、M8-18B。
- **任务**：记录 `hypothesis_generation`、`hypothesis_validation`、`evidence_collection`、`verification`、`finding_validation` 的摘要；统计有效/无效假设、请求数、成功/失败证据、三态 verdict、token 与时延。
- **非目标**：不存原始敏感证据或完整 reasoning。
- **验收与证据**：成功、空短路、证据失败、预算耗尽均生成一致结构的 trace；统计可由测试重算；敏感信息测试证明 trace 已脱敏。

## M8-20：`single` / `verify` 模式开关

- **目标**：在同一 review service 中显式、可对照地选择单轮或验证模式。
- **依赖**：M8-19。
- **任务**：增加受限 `llm_review_mode`（`single`/`verify`）配置、CLI 参数和 state/report 元数据；两种模式共用 ContextBudget、finding validation、ranker 和输出。
- **非目标**：不修改默认 single 行为；verify 失败不可伪装成 single 成功。
- **验收与证据**：默认与非法值测试明确；两种模式有离线 smoke；模式在 trace/报告可见；verify 失败时结果状态与错误可见且没有未经验证 finding。

## M8-21：误报导向 Eval Cases

- **目标**：建立“表面像问题、但确定性上下文能否定假设”的真实负样例集。
- **依赖**：M8-0；不依赖 controller 完成。
- **任务**：至少新增 2–3 个 case，例如调用方已捕获异常、上层 context manager 管理资源、API 契约保证非空返回、重导出仍提供 import；每个 case 写明单轮可能误报原因、应使用的证据类型和 `must_not_findings`。
- **非目标**：不为 case 名称或固定文件名增加特判，不把负样例当作空 diff。
- **验收与证据**：ground truth 人工核验；case 在缺少额外上下文时确有误报风险；expected 文件表达禁止 finding；重命名 case 不影响逻辑；测试能显示 single 与 verify 的逐 case 差异。

## M8-22：Single 与 Verify 对照 Eval

- **目标**：验证 M8 是降低目标误报，而不是简单拒绝更多问题。
- **依赖**：M8-20、M8-21。
- **任务**：在固定 commit、case、模型、随机性和预算下对照 single/verify；保存逐 case finding、verdict、证据失败与成本数据；只分析结果，不在本轮修改实现。
- **非目标**：不根据结果回调 prompt、阈值或 ground truth。
- **验收与证据**：报告 Precision、FPR、Recall、F1、rejected/inconclusive 数、平均调用数、token、时延、估算成本、证据失败率；逐 case 标识新增误报、消除误报与新增漏报；若 Recall 明显下降或大量问题变 inconclusive，结论必须如实标注为未达标/需进一步实验。

# 三、推荐执行顺序与最小可验收范围

```text
稳定既有 M0–M4（含可复现基线）
        ├─ M7-0
        │   └─ M7-1 → M7-2 → M7-3
        │       ├─ M7-4 / M7-5 / M7-6A / M7-6B（只读工具，按依赖完成）
        │       └─ M7-7 → M7-8 → M7-9 → M7-11
        │           ├─ M7-10（真实 provider 适配，可在 mock happy path 后独立进行）
        │           └─ M7-12A → M7-12B → M7-13 → M7-14 → M7-15 → M7-16
        │               └─ M7-17 → M7-18
        └─ M8-0
            └─ M8-1 → M8-2 → M8-3 → M8-4 → M8-5
                ├─ M8-6 → M8-7 / M8-8 / M8-9 → M8-10 → M8-11
                └─ M8-12 → M8-13
                    └─ M8-14 → M8-15 / M8-16 / M8-17
                        └─ M8-18A → M8-18B → M8-19 → M8-20 → M8-21 → M8-22
```

斜杠表示可以并行开发、但汇合任务必须等待两侧完成；同一行的“/”不是让同一提交同时实现多个任务。M7 和 M8 可共享已验证的底层脱敏、路径安全、预算和最终 finding 校验能力，但控制器、提示词、trace 阶段名与模式语义必须彼此隔离。

### M7 最小可验收版本

按顺序完成：M7-0、M7-1、M7-2、M7-3、M7-5、M7-6A、M7-7、M7-8、M7-9、M7-11、M7-12A、M7-14、M7-15、M7-16、M7-17、M7-18。暂缓并行 tool calls、完整 import 链、复杂恢复、自动规划与多 Agent。

### M8 最小可验收版本

# 四、每项任务的交付记录模板

以下模板用于让下一位实现者无需重新猜测任务边界。每完成一个任务复制一份；方括号内容必须替换为实际事实，未知项保留为“未验证”并说明原因。

```markdown
## [M7-x / M8-x] 交付记录

- 状态：[已完成 / 未完成 / 被阻塞]
- 依赖核验：[依赖任务、对应测试或代码证据]
- 变更范围：[生产代码、测试、Eval、配置、文档]
- 实现契约：
  - 输入：[类型、允许范围、默认值]
  - 输出：[类型、稳定 ID/状态码、截断语义]
  - 失败与降级：[异常、预算耗尽、空结果如何表示；是否会产生 finding]
  - 兼容性：[fixed/single 默认路径和现有 API 是否变化]
- 验收证据：
  - `[实际运行的测试命令]` → [真实结果]
  - `[实际运行的 Eval 命令，如适用]` → [结果文件/关键指标]
  - [手工检查或机器可读 trace 的路径]
- 未验证项与风险：[无 / 明确列出]
- 下一项建议：[依赖已满足的单一任务 ID]
```

在没有这份记录或记录中仍有未解释的失败语义时，不把任务标为完成，也不以“后续控制器会处理”为由跳过本任务的验收。
