# RepoReview Agent 学习与工程化计划

更新时间：2026-07-10

## 1. 项目定位

RepoReview Agent 是一个面向真实 Pull Request 工作流的代码审查 Agent：获取 PR diff 和仓库上下文，结合确定性规则与 LLM 生成可定位、可解释、可验证的审查建议，并将结果回写到代码协作平台。

本计划服务于 AI 应用开发 / Agent 工程 / 后端日常实习。目标不是堆叠 Agent、MCP、前端等名词，而是把项目做成：

- 可运行：新用户能在 5 分钟内完成安装并跑通示例。
- 可集成：能在真实 GitHub PR 中自动触发并回写结果。
- 可解释：每条 finding 有证据、来源、定位和置信度。
- 可评估：有可信评测集、基线和质量/成本指标。
- 可调试：执行步骤、失败、token、时延和成本可追踪。
- 可防护：文件读取、敏感信息、Prompt Injection 和模型调用都有边界。

参考项目与资料：

- [PR-Agent](https://github.com/The-PR-Agent/pr-agent)
- [PR-Agent 自动化与使用方式](https://github.com/The-PR-Agent/pr-agent/blob/main/docs/docs/usage-guide/automations_and_usage.md)
- [Anthropic: Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents)
- [Model Context Protocol](https://modelcontextprotocol.io/docs/getting-started/intro)

## 2. 执行原则

1. 先完成稳定的单流程基线，再讨论多 Agent。
2. 先打通真实 PR 工作流，再做独立前端。
3. 每个能力都必须有自动化测试或 Eval 证明，不能只凭一次 Demo。
4. LLM 输出、diff、代码、PR 描述和评论都属于不可信输入。
5. 每个里程碑以退出条件结束，不为凑天数安排机械任务。
6. 文档随功能更新；只在最终验收时集中整理一次，不单独反复“美化”。

### 3.3 新增的重要能力

旧计划缺失或覆盖不足，新增以下内容：

1. GitHub Provider：从 PR URL 获取元数据、diff、变更文件和提交信息。
2. PR 评论闭环：发布摘要和 inline finding，并支持幂等更新而非重复刷评论。
3. 增量审查：PR 新增提交后，只审查新增变化或明确标注审查范围。
4. Finding 证据校验：校验文件、行号、changed hunk、reason、confidence 和 evidence，不再丢字段。
5. LLM 可靠性：超时、有限重试、fallback、最大调用次数、token/费用预算。
6. 可信 Eval：Precision、Recall、F1、误报率、定位准确率、时延和成本，而不只看 category hit rate。
7. 基线对照：静态规则、单 LLM、Rule + LLM 三种方案使用同一评测集比较。
8. GitHub 集成测试：使用 fixture/mock API 测试 PR 获取、权限错误和评论回写。
9. 最小 API/运行记录：为薄前端和 MCP 复用同一个 service 层。
10. MCP 条件式扩展：核心稳定后再暴露标准工具接口。
11. 多 Agent 实验门槛：只有单 Agent 基线稳定且 Eval 显示明确收益时才实现。

## 4. 当前项目基线

### 已有能力

- 本地 `--diff` 输入和固定 review loop。
- diff 解析、文件上下文、静态规则和可选 LLM。
- `ReviewState`、结构化输出校验、JSON/Markdown 报告。
- Mock/OpenAI provider、Trace、5 个 Eval case 和基础测试。
- repo root 越界防护、上下文字符和额外文件数量限制。

### 优先修复的问题

- 缺少 `pyproject.toml` 和明确 Python/依赖约束。
- README 不完整且存在编码/Markdown 问题。
- Mock finding 与输入无关，不能用于有效测试。
- 输出阶段丢失模型的 `reason/confidence`，并把 confidence 固定为 `1.0`。
- Eval 的正例只检查 expected category 是否为实际结果子集，不惩罚额外误报。
- Trace 没有完整记录 step 时延、模型、token、重试和成本。
- 只能接收人工准备的 diff，不能获取真实 PR 或回写评论。

## 5. 精简优化后的学习计划

计划按 7 个里程碑执行。时间是建议投入，不要求机械对应自然日。

## M0：基线修复与可复现环境（1-2 个工作日）

### 学习目标

理解“可以在自己机器运行”与“别人可以复现”之间的差异，并修复会污染后续评测的基础问题。

### 任务

1. 添加 `pyproject.toml`：
   - 明确 `Python >= 3.10`。
   - 声明运行和开发依赖。
   - 提供 `reporeview` console script。
2. 完善 `.gitignore`，排除 `__pycache__/`、`.pytest_cache/`、本地 trace、密钥和临时报告。
3. 修复 README 编码和代码块，只保留：项目定位、安装、Quick Start、当前限制。
4. 修复 finding 数据链路：
   - `reason/confidence/evidence/source` 从 LLM 输出保留到最终报告。
   - 不再统一写死 confidence。
5. 将固定 Mock 改为可配置 fixture/stub，使测试可以覆盖正常 JSON、坏 JSON、超时和空结果。
6. 修复 Eval 正例误报不受罚的问题。
7. 在 CI 中运行 lint/compile、pytest 和离线 Eval。

### 退出条件

```text
pip install -e .
pytest
reporeview review --diff examples/simple.diff --repo .
python -m src.eval_runner
```

以上命令能在干净环境运行；报告中 confidence/reason 不丢失；Eval 会对多余 finding 计入误报。

## M1：可靠的 diff 与本地仓库输入（2-3 个工作日）

### 学习目标

把输入层从教学样例提升为可处理常见真实变更的稳定组件。

### 任务

1. 为 `ChangedFile` 和 `DiffHunk` 建立明确 schema。
2. 支持 modified、added、deleted、renamed、binary、多 hunk 和带空格路径。
3. 正确维护 old/new line number，避免 finding 指向错误行。
4. 支持直接从本地仓库生成 diff：

```text
reporeview review --repo . --base main --head HEAD
```

5. 对超大文件、缺失文件、子模块和 binary diff 采取明确降级策略。
6. 为每种 diff 类型增加 fixture 和单元测试。

### 退出条件

- 至少覆盖 7 类真实 diff 场景。
- deleted/renamed/binary 文件不会导致上下文读取崩溃。
- finding 行号能映射回新文件 changed line。
- CLI 不再强制用户手工生成 diff 文件。

## M2：上下文检索与 Agent State（3-4 个工作日）

### 学习目标

理解上下文相关性、token 成本、安全边界和可解释性之间的权衡。

### 任务

1. 保持 `ReviewState` 为唯一主状态，补充：

```text
task_id / run_id
review_scope
context_budget
model/provider
raw_findings / final_findings
usage / cost
step_status / errors
```

2. 引入统一 `ContextBudget`：

```text
max_total_tokens 或 max_total_chars
max_file_tokens
max_extra_files
max_llm_calls
```

3. Python 文件使用 `ast` 获取变更行所在函数、方法和类。
4. 上下文优先级：changed hunk → 所在符号 → imports → 同文件调用 → 少量本地依赖。
5. 每段上下文记录 provenance：来源、选择原因、截断状态、字符/token 数。
6. 非 Python 文件保留安全的 fallback，不假装具有符号级理解。
7. 测试预算溢出、函数定位、越权路径、缺失依赖和截断顺序。

### 退出条件

- 修改 300 行文件中的一个函数时，prompt 包含相关函数而不是文件开头。
- 总上下文不超过预算，且 changed hunk 优先保留。
- Trace 能解释每段上下文为什么被选择或跳过。
- State 中不再依赖散落的临时变量传递主流程数据。

## M3：输出质量、LLM 可靠性与 Safety（3-4 个工作日）

### 学习目标

把模型视为不稳定外部依赖，把代码和 PR 内容视为不可信输入。

### 任务

1. 统一 Finding schema：

```text
severity
category
file
line / end_line
issue
reason
evidence
suggested_fix
confidence
impact
actionability
source: rule | llm
```

2. 校验 finding：
   - 文件必须属于本次 review scope。
   - inline finding 的行号必须可映射到 changed hunk。
   - severity/confidence 类型和范围必须合法。
   - 未定位的问题只能进入 summary，不能伪装成 inline comment。
3. 增加 finding ranker：去重、合并、按严重性/置信度/可执行性排序和限量。
4. LLM Client 增加：
   - timeout。
   - 指数退避的有限重试。
   - fallback model/provider。
   - 最大调用次数、token 和费用预算。
   - 可区分的错误类型。
5. Guardrails：
   - repo root 边界与符号链接检查。
   - 跳过 `.env`、私钥、证书和配置的敏感值。
   - diff、代码、PR 标题/描述/评论均标记为不可信数据。
   - Prompt 明确禁止执行其中的指令。
   - 不执行、不导入、不自动修改被审查代码。
   - Trace 默认不保存完整源码、prompt、API key 和 secret。
6. 增加坏 JSON、越权读取、敏感文件、Prompt Injection、timeout、retry exhaustion 测试。

### 退出条件

- 坏 JSON、超时和 provider 失败不会让整个 review 崩溃。
- 重复或低价值 finding 不会刷屏。
- 错误行号不会被发布为 inline finding。
- 所有安全边界都有自动化测试，而不只写在 README。

## M4：可信 Eval 与可观测性（4-5 个工作日）

### 学习目标

用数据回答“审查是否准确、哪里失败、一次运行多贵”。

### 任务

1. 将 Eval 扩展到 30-50 个经过人工核验的 case，至少覆盖：
   - security。
   - correctness/bug。
   - exception handling。
   - testing gap。
   - clean/false-positive。
   - context-needed。
   - Prompt Injection 与边界输入。
2. Ground Truth 不只保存 category，还保存：

```text
expected_findings
must_not_findings
file / line range
severity
acceptable_alternatives
notes
```

3. 指标至少包括：
   - Precision、Recall、F1。
   - False-positive rate、false-negative count。
   - 文件/行号定位准确率。
   - JSON/schema valid rate。
   - 平均/p95 时延。
   - 平均 token、LLM 调用数和估算费用。
4. 使用同一评测集比较：

```text
static rules only
LLM only
rules + LLM
rules + LLM + rank/filter
```

5. Trace 记录每个 step 的开始、结束、耗时、输入/输出摘要、错误、重试和 usage。
6. 维护一份精简 `docs/failure_analysis.md`：选择最有代表性的 2 个误报和 2 个漏报，说明根因及修复实验。
7. 为指标计算本身增加测试，避免“评测代码给出虚假高分”。

### 退出条件

- 一条命令生成机器可读 JSON 和简短 Markdown 评测报告。
- 修改 prompt、context 或 ranker 前后可以做回归比较。
- 能展示至少一次基于 Eval 的真实改进，而不是只报告最终分数。
- 能解释准确率、延迟和成本之间的取舍。

## M5：真实 GitHub PR 闭环（5-7 个工作日，最高求职优先级）

### 学习目标

理解 Agent 如何进入真实开发工作流，包括平台 API、权限、幂等性、失败恢复和增量事件。

### 任务

1. 抽象 `GitProvider`，先只实现 GitHub：
   - 输入 PR URL 或 owner/repo/number。
   - 获取标题、描述、base/head SHA、changed files、patch 和提交信息。
2. 将核心 review loop 抽成与 CLI/GitHub 解耦的 service。
3. 支持 GitHub Action 的 `pull_request` opened、reopened、synchronize 事件。
4. 先发布一个 summary/check，再发布通过定位校验的少量 inline comments。
5. 评论必须幂等：更新已有 bot comment，避免每次 push 重复刷屏。
6. 记录 reviewed SHA；PR 更新时明确采用增量审查还是全量重审。
7. 限制 GitHub token 权限，密钥只从 Actions Secrets/环境变量读取。
8. 对 fork PR、权限不足、rate limit、patch 缺失和 API 失败给出明确降级。
9. 使用 mock HTTP/GitHub fixture 编写集成测试：
   - 获取 PR 成功。
   - 权限失败。
   - 评论创建和更新。
   - inline 行号无效时降级到 summary。
10. 在自己的测试仓库完成一次真实 PR Demo。

### 退出条件

- 打开或更新测试 PR 会自动触发 review。
- PR 中能看到结构化摘要和少量有效 inline finding。
- 重跑 workflow 不会产生重复评论。
- API key/token 不进入日志和仓库。
- Demo 不是只上传 artifact，而是完成真实评论回写闭环。

## M6：产品化展示与可选扩展（3-5 个工作日）

### A. 必做：最小展示层

1. 提供最小 HTTP API：

```text
POST /reviews
GET /reviews/{run_id}
GET /reviews/{run_id}/findings
```

2. 使用 SQLite 或简单持久化保存 run 状态、finding、usage 和用户反馈，不保存密钥和完整敏感源码。
3. 做一页薄前端或服务端页面：
   - 输入 PR URL。
   - 显示运行步骤和状态。
   - 展示 diff/文件、finding、severity、evidence、时延和费用。
   - 支持标记“有帮助/误报”。
4. 不做登录、复杂权限、多租户、视觉动效和管理后台。

### B. 可选：MCP Server

仅在核心 service 和 API 稳定后实现，工具建议：

```text
review_pr
review_diff
get_review_result
explain_finding
```

要求 MCP、CLI、API 和 GitHub Action 复用同一 service，不能复制业务逻辑。为 tool schema、错误返回和敏感信息边界编写测试。

### C. 实验：多 Agent

仅在满足以下条件时开始：

- 单 Agent 基线已经在固定 Eval 上稳定。
- 已经明确某类问题需要独立上下文或提示策略。
- 可以记录每个 Agent 的时延、token、费用和输出来源。
- 有单 Agent 与多 Agent 的 A/B 对照。

可尝试并行 specialist reviewer：correctness、security、performance，再由确定性 aggregator 去重排序。默认不要增加自由规划、handoff 或无限反思循环。

只有当多 Agent 在 Recall/F1 上有稳定提升，且误报、时延和成本可接受时，才进入主路径；否则保留为实验结果。

### 退出条件

- 面试官能在 3 分钟内看懂一次真实 review 的输入、过程、结果和指标。
- MCP 或多 Agent 即使不启用，核心 CLI/GitHub 工作流仍完整可用。
- 所有扩展都能说明解决了什么真实问题，而不是只增加技术名词。

## M7：ReAct 工具调用循环（4-5 个工作日）

### 学习目标

把 Agent 从固定 8 步流水线升级为具备动态决策能力的系统：LLM 可以根据当前已知信息自主决定下一步调用哪个工具（读文件、搜索符号、查 import 链、终止并输出），而不是按预设顺序执行所有步骤。

理解 ReAct（Reasoning + Acting）模式的核心 trade-off：灵活性提升 vs 时延/成本/不可控性增加，以及如何在代码审查这种高精度要求的场景中约束 LLM 的行动空间。

### 任务

1. 定义工具协议（Tool Protocol）：

```text
class ReviewTool(Protocol):
    name: str
    description: str
    schema: dict   # JSON Schema for tool input
    def run(self, arguments: dict) -> ToolResult
```

2. 实现初始工具集（复用已有模块，不重写逻辑）：

```text
read_file_context     # 复用 file_context.read_file_context
search_python_symbol  # 复用 file_context.locate_python_symbol
get_changed_hunks     # 复用 diff_parser 解析结果
list_imports          # 复用 file_context._extract_import_file_candidates
finish_review         # 终止循环，输出最终 findings
```

3. 改造 LLM Client 支持 function calling：
   - OpenAI provider 使用 `tools` 参数传递工具 schema。
   - Mock provider 支持可配置的工具调用序列（如 read_file → search_symbol → finish_review）。
   - 每轮 LLM 响应解析出 `tool_calls` 或 `finish`。

4. 实现 ReAct 循环控制器：

```text
while not finished and steps < max_steps:
    response = call_model(prompt + tool_history)
    if response.is_finish:
        break
    for tool_call in response.tool_calls:
        result = dispatch_tool(tool_call)
        tool_history.append(tool_call, result)
    update_trace(step, tool_calls, results)
```

5. 约束与 Guardrails：
   - `max_steps`（默认 5，上限 10）防止无限循环。
   - `max_llm_calls` 与 `max_total_tokens` 仍然生效。
   - 工具只能读取 `repo_root` 内的文件，复用越权检查。
   - 工具结果经过敏感文件脱敏后再进入 tool_history。
   - LLM 无法调用未注册的工具，schema 校验失败时记录错误并跳过。

6. 在 `ReviewState` 中记录 ReAct 轨迹：

```text
react_steps: [
    {step: 1, reasoning: "...", tool: "read_file_context", args: {...}, result_summary: "...", duration_ms: ...}
]
```

7. Eval 对照实验：
   - 新增 eval case：变更涉及跨文件依赖的场景（需要 LLM 主动读取额外文件才能发现的问题）。
   - 使用同一 eval set 对比：固定 pipeline（当前 M0-M6）vs ReAct 循环。
   - 记录两组的 Precision/Recall/F1/时延/成本/token 数。

8. 测试：
   - Mock 路径：验证循环能在 `finish_review` 后正确终止。
   - Mock 路径：验证 `max_steps` 耗尽时能安全降级为当前已有 findings。
   - Mock 路径：验证工具调用参数非法时不会崩溃。
   - 验证 trace 中每个 ReAct step 都有 reasoning、tool、result 和耗时。

### 退出条件

- 同一个需要跨文件上下文才能发现的 bug（如调用了已删除的函数），ReAct 模式能通过工具调用发现，而固定 pipeline 漏报。
- ReAct 循环不会出现无限调用、越权读取或 trace 缺失。
- Eval 对照表能展示 ReAct 在 Recall 上是否有提升，以及时延/成本的增幅是否可接受。
- Mock 模式下可以完全离线测试 ReAct 循环，不依赖真实 API。
- 简历可以写出：「将固定步进流水线升级为 ReAct 工具调用循环，LLM 可自主调用文件读取、符号定位和 import 分析工具，在需要跨文件上下文的 eval case 上将 Recall 从 X 提升到 Y，同时通过 max_steps 和 token 预算约束成本。」

## M8：多轮 LLM 对话与假设-验证审查（3-4 个工作日）

### 学习目标

把 LLM 审查从单次 prompt-response 升级为多轮对话：第一轮 LLM 提出审查假设（"这里可能缺少异常处理"），系统自动收集验证证据（读取相关代码、检查调用链），第二轮 LLM 根据证据确认或否定假设，最终只输出经过验证的 findings。

理解多轮对话在 AI Agent 中的价值：不是让 LLM 自言自语，而是让系统在 LLM 和工具之间建立反馈闭环，用确定性代码验证 LLM 的猜测，降低误报。

### 任务

1. 定义多轮对话状态：

```text
@dataclass
class DialogueTurn:
    role: str           # "reviewer" | "verifier" | "system"
    hypothesis: str      # LLM 提出的待验证假设
    evidence_needed: list[str]  # LLM 请求的证据类型
    evidence_collected: list[dict]  # 系统返回的验证结果
    verdict: str         # "confirmed" | "rejected" | "inconclusive"
    finding: Optional[ReviewIssue]
```

2. 实现假设生成 prompt（第一轮）：
   - 输入：diff、上下文、规则结果。
   - 输出：假设列表，每个假设包含 file/line/issue/evidence_needed。

```text
{
  "hypotheses": [
    {
      "file": "app.py",
      "line": 10,
      "hypothesis": "缺少异常处理",
      "evidence_needed": ["call_chain", "exception_handling_in_caller"],
      "confidence": 0.6
    }
  ]
}
```

3. 实现证据收集器（确定性代码，非 LLM）：
   - `call_chain`：提取被调用函数的定义和异常处理情况。
   - `exception_handling_in_caller`：检查调用方是否有 try/except。
   - `import_chain`：追踪 import 依赖链是否引入风险。
   - 每个证据收集器复用 M2 的 file_context 和 AST 能力。

4. 实现验证 prompt（第二轮）：
   - 输入：原始假设 + 收集到的证据。
   - 输出：每个假设的 verdict 和（如果 confirmed）完整 finding。

```text
{
  "verdicts": [
    {
      "hypothesis_index": 0,
      "verdict": "confirmed",
      "finding": { ... },
      "reasoning": "被调用函数 open() 可能抛出 OSError，调用方无 try/except"
    }
  ]
}
```

5. 多轮对话控制器：

```text
turn_1 = call_model(hypothesis_prompt)    # 生成假设
evidence = collect_evidence(turn_1.hypotheses)  # 确定性收集
turn_2 = call_model(verification_prompt, evidence)  # 验证假设
final_findings = [v.finding for v in turn_2.verdicts if v.verdict == "confirmed"]
```

6. 成本控制与降级：
   - 如果 LLM 第一轮返回空假设列表，直接结束，不发起第二轮调用。
   - 如果证据收集失败（文件不存在、AST 解析失败），标记假设为 `inconclusive`，不输出 finding。
   - 第二轮验证后的 confirmed findings 仍然经过 M3 的 validation（行号校验、severity 修复等）。
   - 总 token 和调用次数受 `ContextBudget` 约束。

7. Trace 记录多轮对话全过程：

```text
dialogue_turns: [
    {
        turn: 1,
        type: "hypothesis_generation",
        hypotheses_count: 3,
        duration_ms: ...,
        tokens: ...
    },
    {
        turn: 2,
        type: "evidence_collection",
        evidence_collected: [...],
        duration_ms: ...
    },
    {
        turn: 3,
        type: "verification",
        confirmed: 2,
        rejected: 1,
        duration_ms: ...,
        tokens: ...
    }
]
```

8. Eval 对照实验：
   - 使用 M4 的 eval set 对比：单轮 LLM vs 多轮假设-验证。
   - 重点关注 false positive rate 是否下降（多轮验证的核心价值是减少误报）。
   - 记录时延和成本增幅。

9. 测试：
   - Mock 路径：假设生成返回 2 个假设，验证后 1 个 confirmed、1 个 rejected。
   - Mock 路径：假设生成返回空列表，验证轮不发起调用。
   - Mock 路径：证据收集异常，假设标记为 inconclusive。
   - 验证 confirmed findings 仍经过行号和 severity 校验。
   - 验证 trace 记录了完整的多轮对话信息。

### 退出条件

- 同一个 eval set 上，多轮验证的 false positive rate 低于单轮 LLM。
- 被拒绝的假设不出现在最终 findings 中。
- 证据收集失败不会导致崩溃或输出未验证的 finding。
- Mock 模式下可以完全离线测试多轮对话流程。
- Eval 对照表能展示多轮对话在 precision 上的提升和成本增幅。
- 简历可以写出：「实现假设-验证两轮 LLM 对话机制，第一轮生成审查假设，系统用 AST 和调用链分析收集确定性证据，第二轮验证后只输出 confirmed findings，在 eval set 上将 false positive rate 从 X% 降至 Y%，代价是平均增加一次 LLM 调用。」

## 6. 最终验收清单

### 核心工程

- [ ] 干净环境可以安装，Python 和依赖版本明确。
- [ ] 本地仓库和真实 GitHub PR 两种输入都能运行。
- [ ] 支持常见 diff 类型和正确行号映射。
- [ ] Context Budget、AST 函数级上下文和 provenance 可验证。
- [ ] ReviewState 包含完整输入、中间状态、usage、错误和输出。
- [ ] Finding schema 保留 reason/confidence/evidence/source。
- [ ] LLM 有 timeout、有限重试、fallback 和调用/费用预算。
- [ ] 去重、过滤和定位校验能减少噪声与错误评论。

### 质量与安全

- [ ] 30-50 个 Eval case 有人工核验 Ground Truth。
- [ ] 有 Precision/Recall/F1/FPR、定位、时延和成本指标。
- [ ] 有至少一次基于 Eval 的前后对照改进。
- [ ] Trace 能定位每个核心 step 的失败和耗时。
- [ ] 越权读取、敏感文件、Prompt Injection 和坏模型输出都有测试。
- [ ] 默认不在 Trace/日志中保存密钥、完整敏感代码或 prompt。

### 真实工作流与展示

- [ ] GitHub PR 打开/更新会自动触发 review。
- [ ] 摘要和 inline finding 能幂等回写。
- [ ] README 只保留一条最短 Quick Start、一张架构图、一次真实 Demo 和一组 Eval 结果。
- [ ] Failure Analysis 展示代表性的误报和漏报。
- [ ] 可用 30 秒说明价值、3 分钟说明架构和取舍。

## 7. 建议的简历表述形成条件

只有完成 M0-M5 后，才适合把项目描述为真实 PR Review Agent。可使用的数据必须来自实际 Eval 和运行记录，不写无法证明的“显著提升”。

示例结构：

```text
实现面向 GitHub Pull Request 的代码审查 Agent，构建 diff 解析、AST 函数级上下文检索、规则与 LLM 分层审查、结构化校验、finding 去重排序和评论回写闭环；设计包含 X 个真实/合成 case 的离线评测集，统计 Precision、Recall、定位准确率、时延与单次成本，并通过上下文预算和质量过滤将某项实测指标从 A 改进到 B；针对越权文件读取、敏感文件、Prompt Injection 和模型失败实现可测试 Guardrails。
```

不要在尚未完成时写：

- “多 Agent 协同显著提升准确率”。
- “支持企业级大规模仓库”。
- “实现完整 MCP 生态”。
- “达到生产级安全”。

项目是否摆脱“玩具感”，最终取决于真实 PR 闭环、可信评测、失败处理和可复现性，而不是页面数量、Agent 数量或协议数量。
