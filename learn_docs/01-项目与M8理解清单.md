## 三、M8 的核心心智模型

M8 不是“再加一次 LLM 调用”，而是把以下不安全链条：

```text
模型猜测 → 直接 finding
```

替换成：

```text
第一轮：不可信模型输出有限 hypothesis
  → 本地 schema / 范围 / 数量校验
  → 确定性 collector 获取白名单 evidence
  → 第二轮：只根据 hypothesis + evidence 输出三态 verdict
  → 只有 confirmed candidate 进入既有 validation / reporter
```

### M8 中每个对象的职责

| 对象 | 它是什么 | 它绝不是什么 |
| --- | --- | --- |
| `ReviewHypothesis` | 待验证的、带稳定 ID 的怀疑 | 最终 finding |
| `EvidenceRequest` | 被白名单限制的确定性查询请求 | 任意读取/搜索/执行授权 |
| `EvidenceItem` | collector 得到的事实及状态 | 模型自行推导的结论 |
| `VerificationVerdict` | 对一个 hypothesis 的 `confirmed/rejected/inconclusive` 判断 | 绕开 validation 的报告内容 |
| `ReviewIssue` | 通过既有校验的最终报告实体 | 任何一轮 LLM 的原始文本 |

### 必须反复确认的语义

- [ ] `not_found`：按请求范围查询后没有找到，是已知事实。
- [ ] `unavailable`：系统未能获取证据（解析失败、权限、超时等），不是反证。
- [ ] `truncated`：信息不完整，不能支持 `confirmed`。
- [ ] `conflicting`：证据之间冲突，默认应走 `inconclusive`。
- [ ] `rejected`：已有证据否定假设，绝不生成 finding。
- [ ] `inconclusive`：证据不足或系统无法验证，绝不生成 finding；但要在 Eval 中统计，避免用它掩盖 Recall 下降。

## 四、按 M8 任务阶段需要搞懂什么

### 阶段 A：基线、协议、第一轮假设（M8-0 ～ M8-5）

| 任务 | 开始前必须理解 | 完成后你应能讲清 |
| --- | --- | --- |
| M8-0 | `single` 现有输出、Eval ground truth、Precision/Recall/F1/FPR | 基线为何必须在固定 commit、case、配置、预算下保存，且不能和 M7 对照混用。 |
| M8-1 | `ReviewIssue` 与 JSON-safe dataclass 的边界 | hypothesis/evidence/verdict 为什么要分开建模；稳定 ID 为何比列表下标可靠。 |
| M8-2 | 现有工具安全边界与参数 schema | 白名单、数量上限、去重键如何避免模型无限探索。 |
| M8-3 | LLM JSON 是不可信输入 | 坏 JSON、部分非法、超限、空列表分别如何处理；被拒项为何不能流入下一阶段。 |
| M8-4 | ContextBudget、prompt 注入边界 | 为什么 prompt 只允许“提出假设”，不能请求任意文件或执行代码。 |
| M8-5 | provider timeout/retry/mock 行为 | 合法空结果、坏输出和 provider 故障为什么是三种不同语义。 |

### 阶段 B：确定性证据（M8-6、M8-8、M8-10、M8-11）

### 阶段 C：第二轮验证与最终出口（M8-12 ～ M8-17）

- [ ] 能解释第二轮 parser 为什么要检查 verdict ID 的覆盖、唯一性和合法状态。

## 五、每次让 Agent 开发前的五个问题

1. 本轮对应哪个**单一** M8-x？它的依赖已经由代码/测试确认了吗？
2. 输入与输出是什么？哪些输入是不可信的？稳定 ID、状态码、截断语义在哪里定义？
3. 最终 `ReviewIssue` 会经过哪条既有 validation 链？`rejected/inconclusive` 如何被保证不越过它？
4. 正常、空、坏 JSON、超时、预算耗尽、collector 失败各自的最终行为和 trace 是什么？
5. 哪个离线测试和哪条 Eval 证据能证明本轮真的完成？
