1. 你这个项目为什么叫 Agent，而不是普通脚本？
普通的脚本里面是写死的固定逻辑，我现在的agent把任务拆解成了receive_task, parse_diff, collect_context, run_static_checks, run_llm_review, validate_output, render_report, save_trace这些状态
后面打算把固定步骤升级成根据state动态选择action的loop

后续再把某些步骤升级成可以让模型决定的

4. 你怎么防止 Agent 无限循环或者乱调用工具？
后续增加max_steps, allowed_tools, trace和错误状态
每一步都记录step name, 输入摘要、输出摘要
超过max_steps或者连续失败，就停止并输出失败的原因
现在是固定八步

5. 为什么要有 trace_steps？它的价值是什么？
为了让agent的执行过程可解释、可调试、可评估
一次review不好知道是diff没有解析对，上下文没有读到，规则没有命中
后续做eval时，可以统计每一步的好事，是否调用LLM，LLM prompt是否稳定

6. 你的 rule-based reviewer 和 LLM reviewer 分工是什么？
rule-based reviewer负责稳定、便宜、确定性的检查，去匹配print、TODO、hardcoded token、测试缺失
LLM reviewer负责更加语义化的问题，比如说逻辑边界、异常处理、跨文件影响和潜在bug
所有的检查都交给LLM，成本会很高，如果只靠规则，发现不了复杂语义性的问题

10. 固定步骤 loop 和动态 Agent loop 的区别是什么？
固定loop是人提前定义流程，动态loop是agent根据当前的state来决定下一步：
看到上下文不够在读一个文件
看到测试缺失，搜索测试目录
看到输出不合法，修复json
动态loop的表达能力更强，也更加容易失控，所以要加max_steps, allowed_tools, trace、结构化校验和错误处理
固定loop的意义是把基础链路做稳定，然后再把上下文检索、额外文件读取这些内容做成动态action
