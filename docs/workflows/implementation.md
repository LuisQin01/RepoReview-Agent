# 实施工作流

本文件是 RepoReview Agent 所有实施任务的长期规则。它规定工作方式，不替代任务卡；任务卡提供本轮唯一目标、非目标、依赖、代码位置和退出条件。`prompts/01-实施任务.md` 只是把这些任务信息填入一次任务请求，不能降低本文件或 `AGENTS.md` 的要求。

## 1. 修改前核验

开始修改前必须：

1. 完整读取当前适用的任务卡；若没有任务卡，先写清本轮目标、非目标和退出条件。
2. 读取相关生产代码、现有测试、配置和实际入口，并沿真实调用链检查输入、状态、输出、错误和副作用。不得根据计划、标题或文档假设能力已经存在。
3. 执行并记录 `git status --short` 和 `git diff`，保存实施基线。识别并排除用户已有的无关改动，不覆盖、回滚或顺手整理它们。
4. 区分已确认事实、待验证假设、受影响文件、明确不改的表面和主要风险。缺少输入/输出契约、失败语义、权限边界或预算归属时，先补证据再动手。

## 2. 范围与设计

- 每轮只完成一个边界清楚、可独立验证的小任务；不主动实现后续功能、平台、前端、MCP、多 Agent 或用户系统。
- 选择最小、直接、可测试、可回滚的改动。不得无理由重构无关代码，也不要为了“像 Agent”添加不必要的框架、抽象或依赖。
- 先定义输入、输出、状态/预算所有者，以及空输入、非法输入、上限、截断、超时、解析失败、耗尽和无结果的行为。保持已有接口、字段、数据结构和默认路径兼容；新增可选行为必须显式 opt-in。
- 复用已有 schema、service、验证、脱敏、敏感文件检查、报告和错误类别，不复制平行逻辑。跨层边界只传内部 dataclass 或 JSON 可序列化数据。
- 将 diff、代码、PR 元数据、评论、provider 响应和 LLM 输出当作不可信输入；路径必须相对 `repo_root` 且拒绝绝对路径、遍历、符号链接逃逸和 review scope 外文件。

## 3. 实施禁区

不得：

- 写死动态结果、case 输出、指标、token、费用或测试期望；
- 修改测试名、降低断言、跳过真实入口或只测试孤立函数；
- 吞掉异常、用 `except Exception` 掩盖未知错误、把失败返回空结果，或用错误 fallback 制造虚假成功；
- 暴露 token、API key、敏感源代码、原始内部错误或未脱敏 trace；
- 为绕过失败删除失败路径、禁用验证或改变无关默认行为。

## 4. 测试与证据

新增行为必须有有效的离线确定性测试、Eval 或可复现命令。至少检查正常路径、关键边界和失败/降级路径，并验证真实入口确实到达改动，而不是只证明新函数能运行。涉及外部服务时使用 mock/fake，不在单元测试中调用真实 OpenAI、GitHub 或其他在线服务。

按 `AGENTS.md` 运行适用的聚焦测试；实施任务完成前运行 `py -m pytest tests/ -v`。影响 Eval 时还运行 `py -m src.eval_runner --cases evals/cases --repo .`，并核对逐 case、指标、误报/漏报和退出条件。进程退出码为 0 或测试通过不等于任务完成：还必须证明真实调用链、失败语义和任务退出条件成立。未运行或未证实的内容必须标记为“未验证”。

## 5. 实施后自检与交付

修改后先查看限定范围的 `git diff`，再按 `acceptance.md` 自检：范围、兼容性、安全、幂等性、正常/边界/失败路径、测试真实性和 Eval 证据均要有依据。交付报告至少包含：

- 实际修改的文件和未修改的相关表面；
- 从入口到输出/副作用的真实调用链；
- 实际执行的命令及真实结果；
- 退出条件逐项证据；
- 超时、耗尽、解析失败、provider 失败和无结果时的失败语义；
- 未验证项目、剩余风险、任务 ID、commit 或工作树状态；
- 下一项独立且边界清楚的任务。

若发现超出本轮范围的问题，只记录为后续任务，不借机扩大 diff。
