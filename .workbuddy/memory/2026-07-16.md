## P1-1 / P1-2 耦合性分析（续）

用户问能否并行修。读代码确认：两者是同一条 category 数据流的多个断点，强耦合，不能并行，须作为一个连贯任务一次性打通。

数据流：expected_findings → `_build_react_provider`(eval_runner.py:644) 构造 finish_findings → `finish_review` schema(review_tools.py:286, additionalProperties:False) 校验 → `parse_llm_response`(llm_reviewer.py:225 硬编码 category="llm") → ReviewIssue。

断点：
- P1-2: `_build_react_provider` 构造 finish_findings 时不传 category
- 共享点: `finish_review` schema 不允许 category（P1-1 和 P1-2 都要改这处，两分支并行必然合并冲突）
- P1-1: `parse_llm_response` 硬编码 "llm"，不读候选 category；`_validated_issue` 委托 parse 故连带

单独修任一会回归：只修 P1-2 不动 schema → candidate 带 category 被 additionalProperties:False 拒绝 → finding 丢失（react 更差）；只修 P1-1 不修 P1-2 → candidate 无 category 可保留。

建议修复顺序（数据流方向）：schema 加 category 字段（optional，非 required 以保持 fixed 路径兼容）→ `_build_react_provider` 传 `"category": ef["category"]` → `parse_llm_response` 读候选 category、缺失时 fallback "llm"。`_validated_issue` 委托 parse 无需单独改。

未验证点：`validate_llm_response` 是否也丢弃 category 字段，修 P1 时需确认（不影响"不可并行"结论）。
