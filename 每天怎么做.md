你现在真正缺的不是“再来一份计划”，而是一个**每天不会卡死的执行系统**。我建议你把三件事分层：

1. **Agent 项目：主线**，直接服务找实习。
2. **算法题：底线**，保持面试能力。
3. **科研任务：保底推进**，避免研究生任务积压造成心理压力。

从今天 **2026-06-16** 开始，到 **2026-07-16**，你不要再追求“每天三件事都大推进”。你要追求的是：**每天至少有一个主推进，另外两件事不断线。**

**一、优先级怎么排**

你的总目标是找 Agent 开发实习，所以优先级应该是：

```
第一优先级：RepoReview Agent 项目
第二优先级：LeetCode Hot100
第三优先级：科研任务稳定推进
```

但这不等于每天只做 Agent。更合理的是：

```
每周 5 天学习
3 天 Agent 主攻
1 天 算法主攻
1 天 科研主攻 + 项目整理
```

这样你不会每天都在三件事之间来回切换，把注意力磨没。

**二、每周时间安排**

如果每天只有 2 小时，按这个最低版本执行：

| 星期 | 主任务      | 时间分配                                                |
| ---- | ----------- | ------------------------------------------------------- |
| 周一 | Agent 项目  | Agent 90 分钟 + 算法 30 分钟                            |
| 周二 | Agent 项目  | Agent 90 分钟 + 科研 30 分钟                            |
| 周三 | 算法主攻    | 算法 90 分钟 + Agent 30 分钟                            |
| 周四 | Agent 项目  | Agent 90 分钟 + 算法 30 分钟                            |
| 周五 | 科研 + 整理 | 科研 60 分钟 + Agent README/复盘 40 分钟 + 算法 20 分钟 |

如果能学 4-5 小时，就加到：

```
Agent：2 小时
算法：1 小时
科研：1 小时
复盘/README：30 分钟
```

注意，**算法和科研不是每天都要大块推进**。你现在最怕的是“今天三个都想做，结果三个都没做”。

**三、Agent 项目应该怎么用 Codex 推进**

你要把 Codex 当成一个“结对工程师”，不是当成搜索引擎，也不是让它直接替你做完。每天流程固定成 5 步。

**Step 1：你先给 Codex 今天的目标**

每天开头直接这样问：

```
我今天要继续实现 RepoReview Agent。当前目标是：实现 diff parser，能从 git diff 中解析出文件名、增删行和 patch 内容。

请你先帮我：
1. 拆成 3-5 个小任务；
2. 告诉我需要创建或修改哪些文件；
3. 给出最小可运行版本；
4. 写完后带我运行一个测试样例。
```

关键是不要问：

```
我该怎么学 Agent？
```

而是问：

```
今天这个功能怎么落地？
```

**Step 2：让 Codex 先读项目，再动手**

每次都要求它先看代码：

```
请你先阅读当前项目结构和相关文件，不要急着改代码。读完后告诉我：
1. 当前项目已经有什么；
2. 今天的功能应该接在哪个位置；
3. 最小实现方案是什么。
```

这样可以避免 Codex 乱写一套和你项目不兼容的代码。

**Step 3：让 Codex 实现最小版本**

你不要让它一上来做完整工程。每天只要一个可运行小版本：

```
请按最小可运行版本实现，不要过度设计。实现后请给我一个命令，让我可以验证今天的功能确实成功。
```

例如当天目标是 diff parser，验收标准就是：

```
输入 examples/simple.diff
输出 changed_files、added_lines、deleted_lines、patch
```

**Step 4：让 Codex 带你验证**

每次写完后继续让它做：

```
请运行测试或示例命令，检查是否真的可用。如果报错，请你定位原因并修复。最后告诉我今天实现的功能入口在哪里。
```

你一定要把 Codex 用到“跑起来”这一步。只生成代码不验证，学习效果会很差。

**Step 5：让 Codex 帮你写复盘**

每天最后发：

```
请根据今天的改动，帮我写一段学习复盘，包括：
1. 今天实现了什么；
2. 用到了哪些 Agent 开发知识；
3. 这个功能面试时可以怎么讲；
4. 明天最自然的下一步是什么。
```

这一步很重要。它会把“写代码”转成“能面试表达”。

**四、RepoReview Agent 的每日实现顺序**

你接下来不要再纠结项目怎么拆，按这个顺序来。

**第 1 周：最小可运行**

Day 1：项目骨架 + README
Day 2：diff parser
Day 3：读取被改文件上下文
Day 4：rule-based reviewer
Day 5：生成 Markdown review report

这一周先不急着接 LLM。你要先理解：Agent 项目不是只有大模型，外面的工程壳更重要。

**第 2 周：接入 LLM 和结构化输出**

Day 6：设计 prompt，输入 diff + context
Day 7：让 LLM 输出 JSON review
Day 8：用 Pydantic/dataclass 校验输出
Day 9：坏 JSON 自动重试一次
Day 10：保存 trace 日志

这一周你能讲的 Agent 能力是：

```
tool use、structured output、trace、error handling
```

**第 3 周：上下文和评测**

Day 11：根据 import/function name 找相关文件
Day 12：限制上下文长度
Day 13：准备 10 个 eval case
Day 14：实现 eval runner
Day 15：记录成功率、误报、失败案例

这一周开始有实习竞争力，因为很多人只做 demo，不做 eval。

**第 4 周：包装成作品**

Day 16：多 reviewer：Correctness/Test/Maintainability
Day 17：CLI 命令打磨
Day 18：README 补架构图和示例
Day 19：写 failure analysis
Day 20：整理简历描述和面试稿

**五、每天不知道怎么实现时，直接用这个 Prompt**

你可以每天复制这一段给 Codex：

```
我正在做 RepoReview Agent，一个本地代码审查 Agent。今天的任务是：。

请你作为结对工程师帮我完成：
1. 先阅读当前项目结构和相关代码；
2. 判断今天任务应该改哪些文件；
3. 把任务拆成最小实现步骤；
4. 直接实现最小可运行版本；
5. 添加一个简单示例或测试；
6. 运行验证；
7. 最后用 5 句话解释今天我应该学会什么。

要求：
- 不要过度设计；
- 优先保证能运行；
- 每次只实现今天这个功能；
- 如果有多个方案，选最适合一个月内找实习展示的方案。
```

这段就是你的“每日启动器”。

**六、算法题怎么安排**

算法不要和 Agent 抢主线。你每天只要保持不断线：

```
Agent 主攻日：1 道 Hot100
算法主攻日：3 道 Hot100
周五：复盘错题
```

每题只记录三行：

```
题型：
关键思路：
我错在哪里：
```

不要写长题解。你现在需要的是恢复手感，不是写算法教材。

**七、科研任务怎么避免拖垮你**

科研每天不用都深度做，但必须维护一个“下一步清单”。

每天科研只问自己：

```
我的论文/实验/导师任务，下一步最小动作是什么？
```

比如：

```
读 1 篇论文摘要和结论
整理 1 个实验结果表
改 1 段论文文字
跑 1 个脚本
列 3 个问题问导师
```

科研主攻日再集中做 1 小时以上。这样它不会在心理上一直追着你跑。

**八、你和 Codex 的分工**

你负责：

```
决定今天做什么
判断结果是否符合目标
理解关键代码
把项目讲成面试故事
```

Codex 负责：

```
拆任务
读代码
写最小实现
调试报错
补测试
整理 README
模拟面试问题
```

不要把“理解”也完全外包。正确用法是：**Codex 写 70%，你理解和复述 30%。**

每天结束前，你必须能回答：

```
今天新增的函数入口是什么？
输入是什么？
输出是什么？
失败时怎么处理？
这个功能和 Agent 开发有什么关系？
```

如果答不上来，就让 Codex 解释到你能复述为止。

**九、今天就怎么开始**

今天 2026-06-16，你不要再规划一整晚。就做这件事：

```
创建 RepoReview Agent 的项目骨架和 README 第一版。
```

你可以直接开一个新对话或在当前项目目录里对我说：

```
请帮我从零创建 RepoReview Agent 的项目骨架。先做最小版本：README、src 目录、一个 CLI 入口、examples/simple.diff。今天只要求能运行一个 hello review 命令。
```

然后让 Codex一路做到能运行。

你的重点不是“变得特别自律”，而是建立一个稳定机制：**每天一个小目标，Codex 帮你拆和实现，你负责验收和复盘。**这样坚持一个月，你会留下一个项目、一套算法记录、一个不会彻底失控的科研节奏。
