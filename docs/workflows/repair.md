# P0/P1 修复工作流

本文件规定验收或 review finding 的修复闭环。它只处理已确认成立的 P0/P1，不是扩大范围或顺手重构的入口。`prompts/04-缺陷修复.md` 负责填写一次缺陷的复现信息；本文件负责长期的判断、修复和复验规则。

## 1. 接收与逐项判定

1. 逐项读取验收报告或 review finding，记录稳定编号、级别、文件/位置、触发输入、影响、证据和退出条件关联。
2. 对每项判断为“成立”“部分成立”或“不成立”，并给出可复核的代码、测试或命令证据；不能只因模型或审查者声称存在就修复。
3. 先处理 P0，再处理 P1。P0 必须修复；P1 必须在进入下一个任务前修复。P2 默认只记录，不主动修复；不得为了修 P2 扩大 diff、引入依赖或进行重构。
4. 若根因尚未确认，先复现问题，沿真实调用链提出最多三个可证伪假设并验证；不要直接在最外层补 `try/except Exception`。

## 2. 最小根因修复

- 只修复证据支持的根因，并保持任务卡范围、已有接口/字段/数据结构和默认行为兼容。
- 优先在故障所属层修复：输入解析、状态传递、上下文、规则/LLM、结构化校验、预算控制、报告或平台集成；不要用上层空结果、静默降级、禁用校验或修改 fixture 掩盖下层失败。
- 对失败、超时、解析失败、预算耗尽、provider 不可用和无结果保留稳定且可解释的状态；写入失败不得标记成功，敏感值不得进入日志、trace 或报告。
- 不删除失败路径、不降低断言、不写死动态结果、不制造与真实入口脱节的修复。超出当前任务的发现转为后续独立 task。

## 3. 回归测试与验证

每个成立的 P0/P1 至少增加一个“修复前失败、修复后通过”的离线确定性回归测试，并覆盖相关正常、边界和失败/降级行为。测试应经过真实调用链或最小真实入口；外部服务使用 mock/fake。运行相关聚焦测试，必要时运行完整 `py -m pytest tests/ -v`；影响 Eval 时运行 `py -m src.eval_runner --cases evals/cases --repo .`，检查逐 case 和适用指标而非只看退出码。

修复后重新检查限定范围的 `git diff`，并按照 `acceptance.md` 再次验收：调用链、退出条件、兼容性、安全、幂等性、错误语义和证据必须重新成立。若验证失败，报告真实失败，不返回假成功。

## 4. 修复报告

对每个 finding 报告：

- 编号和原始级别；
- 判定（成立/部分成立/不成立）及证据；
- 根因所在层和最小改动；
- 修改文件、回归测试、实际命令和真实结果；
- 失败/超时/耗尽/解析失败/无结果的语义；
- 是否仍阻塞验收、剩余风险和未验证项。

修复完成后给出新的 `PASS`/`FAIL` 验收结论。只有无 P0、无未修复 P1、退出条件有证据且必要验证真实通过时，才允许进入下一个任务；P2 和超范围问题列为后续任务，不得伪装成已处理。
