# 验收工作流

本文件是统一的只读验收规则。验收服务于项目的学习和求职目标：优先证明核心功能、真实调用链、可信错误处理、真实测试/Eval、可复现的 GitHub PR 闭环和可讲解的 Agent 工程取舍；不以生产级边角优化、复杂框架或代码数量作为完成标准。

## 1. 验收输入与方式

验收者必须先读取任务卡、当前 diff、相关实现和测试，并确认本轮目标、非目标和退出条件。再从真实入口追踪到 service/controller、provider/工具、验证、报告和外部写入（如有）的调用链。验收是只读操作，不修改实现、测试或 fixture。

证据只能来自实际代码、实际命令输出、可复现的离线测试/Eval、受控 mock 集成或记录的运行结果。不得把计划、旧文档中的已勾选项、进程退出码或模型自报结果当成通过证据。未执行、无法定位或与当前数据集不一致的证据均为未验证；只要必要证据未满足，结论就是 `FAIL`。

## 2. 统一检查清单

至少检查：

- 当前任务目标、非目标、退出条件和是否确实只完成一个小边界；
- 真实调用链是否从实际 CLI/API/Eval/任务入口到最终报告或写入，是否存在孤立实现；
- 正常、空/非法输入、边界/上限、截断、超时、解析失败、预算耗尽、provider 失败、无结果和降级路径；
- 自动化测试是否经过真实入口、断言行为而非“能运行”，是否覆盖关键失败路径；Eval 是否固定数据、真实计算逐 case 结果与 Precision/Recall/F1/FPR/定位等适用指标；
- 写死值、吞异常、过宽捕获、错误 fallback、虚假成功、被跳过的失败路径和伪造测试/指标；
- 接口、字段、数据结构、错误码、默认行为和重复运行的幂等性；
- token、API key、敏感代码、PR 内容、prompt、trace、日志和内部错误是否按既有规则脱敏；
- 是否引入不必要依赖、抽象、框架、权限或生产级优化，是否有无关改动；
- Agent 输出是否有稳定 ID、可定位的证据、结构化验证、可解释原因和可复核的最终 `ReviewIssue`；
- 是否体现真正的 Agent 工程能力（受控工具/预算、验证和降级、观测与 Eval），而非仅增加“Agent”命名；
- 结果是否足以写进简历并在面试中解释，且所有数字和能力声明都有证据。

涉及外部 API 写入时，必须验证成功/失败语义和重跑行为：写入失败不能报告成功，重跑不能产生不受控的重复评论或重复副作用。

## 3. 问题优先级

### P0：阻塞问题

包括核心目标未实现、核心路径不可运行、虚假宣称退出条件通过、数据损坏或严重安全问题、token/API key/敏感数据泄露、伪造测试/结果/指标，或改动使项目无法测试、启动或继续开发。P0 必须修复，未修复前不能进入下一个任务。

### P1：重要问题

包括明显影响功能正确性、核心调用链、错误处理/降级、关键测试覆盖、幂等性、数据兼容性、Agent 输出可解释/可定位/可验证性、Eval 真实性、GitHub PR 闭环、简历可信度或面试学习价值的问题。例如：新函数未接入真实入口、只测孤立函数、畸形 LLM 输出中断流程、写入失败却标成功、重跑重复评论、关键字段丢失、失败路径无测试或缺少可复现证据。P1 必须在下一个任务前修复。

### P2：低优先级改进

仅当问题对当前核心功能、Agent 学习、实习竞争力、简历/面试可信度和后续关键里程碑影响较低时，才归为 P2。通常包括格式偏好、轻微命名或重复、非关键注释、极低概率小影响边界、当前阶段用不到的扩展性设计、收益低于成本的生产级优化或代码洁癖。P2 只记录，默认不修复，也不阻止通过；不得借此逃避正确性、调用链、测试真实性、安全、兼容性或项目可信度问题。

## 4. 通过条件与报告

只有同时满足以下条件才可 `PASS`：无 P0、无未修复 P1、每个退出条件都有证据、必要测试真实运行并通过，且没有通过降标准或跳过失败获得通过。否则为 `FAIL`。不使用含糊的“差不多”“条件通过”替代结论；证据不足也为 `FAIL`，并明确缺口。

报告至少使用以下结构：

```markdown
# 验收结论

结论：PASS / FAIL

## P0

- 无，或具体问题、位置、证据、影响和修复要求

## P1

- 无，或具体问题、位置、证据、影响和修复方向

## P2

- 问题
- 为什么属于 P2
- 为什么当前不值得修复
- 是否建议以后处理

## 退出条件核查

- [x] 条件及证据
- [ ] 未满足条件及原因

## 实际验证

- 执行命令
- 真实结果
- 未执行的验证及原因

## 最终判断

- 是否允许进入下一个任务
- 必须先修复的问题
- 可暂缓的问题
```

每个 P0/P1 finding 还应包含稳定编号、文件/位置、触发输入、影响、现有测试为何漏过、最小修复方向和是否阻塞；不得把工具文本、假设或未验证 verdict 直接当作最终 finding。
