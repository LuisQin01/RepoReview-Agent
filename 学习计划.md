# 学习计划

你现在最该做的不是继续“学 Agent”，而是围绕一个能投简历的主项目，把 Hello-Agents 前九章的知识压成工程能力。

我建议从 **今天 2026-06-10 到 2026-07-10**，主线只做一件事：

> **RepoReview Agent：一个本地优先的 Code Review Agent**

它比单纯 LeetCode 复盘 Agent 更适合找实习；LeetCode 复盘可以作为它的一个小场景，而不是主项目。

我刚核对了当前几个方向：OpenAI Agents SDK 把 Agent 能力拆成 agent loop、tools、handoffs、guardrails、sessions、tracing 等工程原语；Anthropic 也强调从简单、可组合模式开始，不要一上来堆复杂框架；LangGraph 的重点是 durable execution、memory、human-in-the-loop 和 tracing；真实开源项目里，PR-Agent、OpenReview、Open SWE 都说明 **coding / review agent 是非常贴近真实岗位的方向**。参考：[OpenAI Agents SDK](https://openai.github.io/openai-agents-python/)、[Anthropic Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents)、[LangGraph Docs](https://docs.langchain.com/oss/python/langgraph/overview)、[MCP Docs](https://modelcontextprotocol.io/docs/getting-started/intro)、[PR-Agent](https://github.com/The-PR-Agent/pr-agent)、[OpenReview](https://github.com/vercel-labs/openreview)、[Open SWE](https://github.com/langchain-ai/open-swe)。

**最终目标**

到 2026-07-10，你要有这些东西：

1. 一个 GitHub 项目：`reporeview-agent`
2. 一个可运行 CLI：输入 `git diff`，输出结构化 code review
3. 支持仓库上下文读取，而不是只看 diff
4. 有 tool calling / 文件工具 / 结构化输出 / trace 日志
5. 有 15-20 条 eval case
6. README 里有架构图、demo、失败案例、改进方向
7. 简历上能写出一段具体项目经历
8. 面试时能讲清楚 Agent loop、RAG/context、tools、eval、guardrails

**每天固定节奏**

你最低每天 2 小时，我建议这样切：

```text
10 分钟：写下今天唯一目标
25 分钟：看文档/教程/源码，只看今天要用的部分
60 分钟：自己实现或改造一个小功能
15 分钟：调试、跑 demo
10 分钟：写复盘：今天做了什么，哪里坏了，怎么修的
```

如果当天能学 4-5 小时，加成这样：

```text
2 小时：Agent 主项目
1 小时：Hello-Agents / 官方文档对照学习
1 小时：LeetCode / 算法恢复
30 分钟：README、复盘、面试表达
```

重点：**每天必须留下代码或文档，不允许只看教程。**

**第 1 周：从跟敲切到独立实现**

时间：2026-06-10 到 2026-06-14  
因为今天是周三，我建议第一周先连续学 5 天，把节奏启动起来。

目标：做出最小版 `RepoReview Agent`。

**Day 1：确定项目骨架**

任务：

1. 建 GitHub 仓库或本地目录。
2. 写 `README.md` 第一版。
3. 明确项目一句话：

```text
RepoReview Agent 是一个本地代码审查 Agent，读取 git diff 和相关文件上下文，输出结构化 review 建议。
```

目录建议：

```text
reporeview-agent/
  src/
    cli.py
    diff_parser.py
    file_context.py
    reviewers.py
    schemas.py
    trace.py
  examples/
  evals/
  docs/
  README.md
```

验收标准：

```text
能用一句话讲清楚项目解决什么问题。
README 里有项目目标、输入输出示例、未来计划。
```

**Day 2：实现 diff 输入**

任务：

1. 支持读取 `git diff` 输出。
2. 解析出变更文件名、增删行、patch 内容。
3. 暂时不用 LLM，先打印结构化结果。

输出格式示例：

```json
{
  "files": [
    {
      "path": "src/auth.py",
      "added_lines": 12,
      "deleted_lines": 3,
      "patch": "..."
    }
  ]
}
```

验收标准：

```text
给一段 diff，程序能识别改了哪些文件。
```

**Day 3：实现文件上下文读取**

任务：

1. 根据 diff 文件路径读取原文件。
2. 支持限制最大读取字符数。
3. 如果文件不存在，要有友好错误。
4. 记录读取了哪些文件。

你今天学到的 Agent 点：

```text
工具不是随便调用的，要有权限、范围、失败处理和日志。
```

验收标准：

```text
输入 diff 后，程序能读取对应文件上下文。
```

**Day 4：做 rule-based reviewer**

先不急着接 LLM，先写规则检查器。

检查项：

1. 是否改了测试但没改业务代码。
2. 是否改了业务代码但没有测试。
3. 是否出现 `TODO`、`print`、`debugger`。
4. 是否可能缺少异常处理。
5. 是否有明显硬编码 key / token 字样。

验收标准：

```text
不用 LLM，也能输出 3-5 类基础 review。
```

**Day 5：接入 LLM reviewer**

任务：

1. 构造 prompt。
2. 输入 diff + 文件上下文 + rule findings。
3. 要求模型输出 JSON。
4. JSON 字段固定：

```json
{
  "findings": [
    {
      "severity": "high",
      "file": "src/auth.py",
      "line": 42,
      "issue": "...",
      "reason": "...",
      "suggested_fix": "...",
      "confidence": 0.78
    }
  ]
}
```

验收标准：

```text
能跑通一次完整 review：diff -> context -> reviewer -> JSON -> markdown。
```

**第 2 周：把 demo 做成 Agent 工程**

时间：2026-06-17 到 2026-06-21  
每周学 5 天，中间两天用于休息、补漏、复盘。

目标：加入 tool calling、trace、结构化校验。

**Day 6：设计 Agent loop**

你要写清楚这几个状态：

```text
1. receive_task
2. parse_diff
3. collect_context
4. run_static_checks
5. run_llm_review
6. validate_output
7. render_report
8. save_trace
```

不要直接上复杂框架，先用 Python 函数组装。

验收标准：

```text
README 有一张流程图或伪代码。
```

**Day 7：结构化输出校验**

任务：

1. 用 Pydantic 或 dataclass 定义 finding schema。
2. LLM 输出不合法时，做一次修复重试。
3. 如果还失败，就保留原始输出并标记失败。

验收标准：

```text
模型输出坏 JSON 时，程序不会直接崩。
```

**Day 8：上下文检索升级**

任务：

1. 不只读取被改文件。
2. 根据 import / function name / class name 读取相关文件。
3. 限制上下文大小，避免塞太多内容。

验收标准：

```text
review 报告里能说明：本次使用了哪些上下文文件。
```

**Day 9：多 reviewer 拆分**

实现 3 个 reviewer：

```text
CorrectnessReviewer：逻辑 bug、边界条件
TestReviewer：测试缺口
MaintainabilityReviewer：可读性、重复代码、复杂度
```

先用三个 prompt 或三个函数，不必真的多 Agent。

验收标准：

```text
最终报告按 reviewer 分类。
```

**Day 10：trace 日志**

每次运行保存：

```json
{
  "task_id": "...",
  "input_files": [],
  "tools_called": [],
  "llm_calls": [],
  "findings_count": 3,
  "duration_ms": 1234,
  "errors": []
}
```

验收标准：

```text
每次 review 后，在 traces/ 下生成一份可读日志。
```

**第 3 周：加入评测，拉开竞争力**

时间：2026-06-24 到 2026-06-28

很多人做 Agent 只做到“能跑”，但实习竞争力来自你能证明它“有多靠谱”。这一周做 eval。

**Day 11：设计 eval case**

准备 10 个小样例：

1. 漏掉空列表判断。
2. 除零风险。
3. 改了函数签名但没改调用方。
4. SQL 拼接风险。
5. 删除异常处理。
6. 改业务逻辑但没加测试。
7. 性能从 O(n) 变 O(n²)。
8. 硬编码配置。
9. 日志泄露敏感信息。
10. 正常代码，无需强行提问题。

验收标准：

```text
evals/cases/ 下有至少 10 个 case。
```

**Day 12：实现 eval runner**

任务：

1. 批量跑 case。
2. 记录每条是否发现目标问题。
3. 输出表格。

指标先简单：

```text
success_rate
json_valid_rate
false_positive_count
average_findings
average_duration
```

验收标准：

```text
一条命令能跑完所有 eval。
```

**Day 13：失败案例分析**

任务：

1. 找 3 个 Agent 没发现的问题。
2. 找 3 个 Agent 误报的问题。
3. 调 prompt 或上下文策略。
4. 写 `docs/failure_analysis.md`。

验收标准：

```text
不是只展示成功 demo，也能讲失败和改进。
```

**Day 14：安全和 guardrails**

任务：

1. 禁止读取项目外文件。
2. 限制最大文件大小。
3. 限制最大 LLM 调用次数。
4. prompt injection 测试：在代码注释里写“忽略之前规则”。

验收标准：

```text
README 里有 Safety / Limitations 一节。
```

**你每天的学习规则**

这部分很重要，因为你说自己学习习惯不好、注意力容易散。

你以后每天开始前，只写一个目标：

```text
今天只完成：diff parser 能识别改动文件。
今天只完成：eval runner 能跑 3 个 case。
今天只完成：README 加上架构图。
```

不要写这种：

```text
今天深入学习 Agent，顺便完善项目，再刷几道题。
```

这会让大脑直接逃跑。

每天结束只回答 4 个问题：

```text
今天做了什么？
今天哪里卡住？
我怎么解决的？
这件事面试时能怎么讲？
```

**判断你是否真的学会**

每学一个知识点，用这个标准验收：

```text
我能不能不用教程重新写出来？
我能不能改一个需求？
我能不能制造一个失败？
我能不能解释为什么失败？
我能不能把它写进 README 或简历？
```

如果答案是否，那就不算学会，只算看过。
