继续。结合你希望节省 token 的目标，我会修正上一版中过于保守的地方：**绝大多数任务不需要 Sol**。官方 Codex 指南也把 Medium 作为交互式编码的通用平衡档，更高推理强度适合多步骤、复杂约束和工具协调任务。

你的任务卡已经拆得很细，每张卡都有明确的目标、非目标、依赖和验收证据，所以 **Terra Medium/High 应该覆盖绝大多数实现**。

## 一、先定义三种验收等级

后面表格中的“验收”使用以下等级：

| 等级            | 执行方式                                                     |
| --------------- | ------------------------------------------------------------ |
| **L0 自检**     | 实现窗口内运行相关测试、检查 diff、填写交付记录，不再发送完整验收提示词 |
| **L1 正式验收** | 实现完成后，在同一窗口发送精简版单任务验收提示词             |
| **L2 独立验收** | 新开窗口，只读取任务卡、diff、相关代码和测试，执行严格验收   |

无论使用哪一档，任务卡要求的真实测试和交付记录都不能跳过。文件明确要求每个任务具备离线测试、真实验证结果、失败语义和移交记录。

# 四、每个小任务是否都要运行两个提示词

## 单任务实现提示词：每个任务都要执行

但不必每次粘贴你那份很长的完整版本。每张任务卡只需发送精简指令：

```text
本轮只完成 M7-3：工具参数 Schema 校验。

读取并遵守 AGENTS.md，以及新增里程碑.md 中 M7-3 的完整任务卡。
先核验 M7-1、M7-2 是否已满足，再实施、测试并填写交付记录。
不要实现 M7-4 或后续任务。
```

基线和 Eval 任务同样属于“单任务执行”，只是它们可能不修改生产代码。

## 单任务验收提示词：不是每个任务都要完整执行

### L0 任务

不需要再发送独立验收提示词，但必须完成：

- 查看 `git diff`；
- 运行任务相关测试；
- 运行任务卡要求的完整命令；
- 填写真实交付记录；
- 确认没有超出非目标。

主要包括：

```text
M7-0、M7-8、M7-17
M8-0、M8-21
```

### L1 任务

在原窗口发送一段精简验收指令即可：

```text
停止新增功能。现在仅验收刚完成的 M7-2。

对照任务卡检查范围、调用链、失败路径、测试质量和真实命令输出。
发现问题只报告，不修改代码。
按 P0/P1/P2/建议分级，并给出通过、有条件通过、不通过或无法验证。
```

不必重新粘贴完整验收清单，因为这些规则应放在 `AGENTS.md` 或专门的审查文档中。

### L2 任务

应该使用完整的严格验收流程，最好新开窗口。新窗口只需要读取：

- `AGENTS.md`
- 当前任务卡
- 当前 diff
- 相关生产代码
- 相关测试
- 实际测试结果或由验收模型重新运行

不需要重新读取所有十个架构文件，除非调用链确实涉及它们。

# 五、推荐的实际节奏

不要机械执行：

```text
新窗口 → 全量初始化 → 实现 → 新窗口 → 全量验收
```

每张任务这样做会浪费大量上下文。

建议按以下节奏：

### 普通任务

```text
原窗口：Terra Medium 实现
→ 测试
→ 同窗口 L0/L1 验收
→ 交付记录
→ 下一任务
```

### 高风险任务

```text
原窗口：Terra High 实现
→ 相关测试 + 全量测试
→ 新窗口：Terra High 独立验收
→ 修复后再次运行测试
```

### 核心架构检查点

```text
Terra High 实现
→ 新窗口 Sol High 验收
```

检查点只保留：

```text
M7-11、M7-18
M8-14、M8-22
```

# 六、一个需要避免的问题

不要让同一条提示词同时要求模型：

1. 实现代码；
2. 验收自己的代码；
3. 发现问题后继续修复；
4. 再自行宣布通过。

这种流程容易使模型为了完成任务而弱化审查。更稳妥的是：

- 实现阶段负责修改和测试；
- 验收阶段明确“只检查，不修改”；
- 验收不通过后，再发送单独的修复提示词；
- 修复完成后只复验原问题和受影响回归。

## 最终执行规则

你可以记成一句话：

> **每个任务都要实现、测试和留下交付记录；只有中高风险任务才需要完整验收，只有 M7-11、M7-18、M8-14、M8-22 值得优先使用 Sol High。**

这样既符合任务清单“一次一个任务、完成后再进入下一项”的要求，也不会把大量额度消耗在简单数据结构、mock 和 Eval case 上。
